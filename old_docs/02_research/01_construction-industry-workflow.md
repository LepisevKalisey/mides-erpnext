# Мировая практика: рабочий флоу строительных компаний

**Статус:** Исследование  
**Дата:** 2026-05-07  
**Источники:** Procore, Sage 300 CRE, Oracle Aconex, CMiC Global, 4castplus, PMI

---

## 1. Стандартный полный цикл строительного проекта

### Фазы (от CRM до закрытия)

```
ФАЗА 1: БИЗНЕС-ДЕВЕЛОПМЕНТ (CRM)
  Лид → квалификация → приглашение на тендер

ФАЗА 2: ТЕНДЕР И СМЕТА (Estimation)
  Получение РД → количественный вынос (Quantity Takeoff)
  → запросы цен у поставщиков (RFQ)
  → формирование себестоимости по статьям затрат (Cost Codes)
  → КП (Bid/Tender)

ФАЗА 3: ПРЕКОНСТРУКЦИЯ (Pre-Construction)
  Победа в тендере → подписание договора с заказчиком
  → конвертация сметы в БЮДЖЕТ ПРОЕКТА
  → подписание договоров с субподрядчиками
  → график производства работ (CPM Schedule)
  → план финансирования (Cash Flow Forecast)

ФАЗА 4: СТРОИТЕЛЬСТВО (Execution)
  4.1 Потребности → закупки → платежи
  4.2 Склад: поступление → выдача в производство → учёт остатков
  4.3 Субподряд: проверка объёма → приёмка (АВР) → контроль оплат
  4.4 Признание расходов для налогообложения
  4.5 Отчёты по использованию материалов и проведённым работам

ФАЗА 5: ЗАКРЫТИЕ (Closeout)
  Финальные акты → исполнительная документация
  → возврат удержания (Retainage Release)
  → получение денег от заказчика
  → постпроектный анализ
```

## 2. Ключевые концепции

### 2.1 Иерархия: Project → WBS → Cost Code → Cost Type

Фундамент финансовой системы. Каждая транзакция тегируется:

```
Project: ЖК «Мирас» (контракт с заказчиком)
  └── WBS (Work Breakdown Structure):
        ├── 01. Земляные работы
        │     ├── Cost Code: 01-100 Разработка котлована
        │     │     ├── Cost Type: Labor (труд)
        │     │     ├── Cost Type: Material (материалы)
        │     │     ├── Cost Type: Equipment (техника)
        │     │     └── Cost Type: Subcontract (субподряд)
        │     └── Cost Code: 01-200 Вывоз грунта
        ├── 02. Фундамент
        │     ├── Cost Code: 02-100 Бетон фундамента
        │     └── Cost Code: 02-200 Монолитные работы
        └── 03. Каркас
```

«Вид работы» в заявке = Cost Code. Каждый Cost Code имеет утверждённый бюджет.  
Без этой привязки нет Job Costing (факт vs бюджет в реальном времени).

---

### 2.2 P2P цикл (Procure-to-Pay) — для закупок материалов

```
[1] Purchase Requisition (PR)
     Сотрудник создаёт заявку на потребность
     Содержит: Project + Cost Code + Qty + Unit + Required Date + оценочная сумма
            ↓
[2] Утверждение ПОТРЕБНОСТИ (Approval #1)
     Руководитель утверждает нужду и оценочный бюджет
     (не цену — цена ещё неизвестна)
            ↓
[3] Sourcing (если поставщик не известен)
     Снабжение отправляет RFQ нескольким поставщикам
     Получает предложения, выбирает лучшее
            ↓
[4] Purchase Order (PO) + Утверждение заказа (Approval #2)
     Официальный документ на закупку с реальной ценой
     Фиксирует: Поставщик + Цена + Условия + Дата
            ↓
[5] Goods Receipt Note (GRN) / Приёмка
     Материалы получены на объекте или складе
     Приёмщик подтверждает факт и количество
            ↓
[6] Three-Way Match (автоматически)
     PR == PO == GRN → ОК → счёт принят к оплате
     Расхождение → флаг для ревью
            ↓
[7] Утверждение оплаты (Approval #3) → Payment
     Казначейство оплачивает
```

**Three-Way Match защищает от:**
- Фантомных поставок (счёт без GRN)
- Откатов (Invoice > PO по сумме)
- Дублированных счётов
- Пересортицы (GRN не совпадает с PO по номенклатуре)
- Двойного списания по проектам

### 2.3 Модель субподряда (Schedule of Values)

```
Договор с монолитчиком на 10 млн ₸
SOV (ведомость объёмов):
  Раздел 1: Опалубка        2 млн (20%)
  Раздел 2: Арматурные р.   3 млн (30%)
  Раздел 3: Бетонирование   4 млн (40%)
  Раздел 4: Распалубка      1 млн (10%)

ЕЖЕМЕСЯЧНЫЙ ЦИКЛ:
  Субподрядчик: «выполнил Раздел 1 на 100%, Раздел 2 на 50%»
  → Подаёт Payment Application (заявка на оплату)
     Сумма: 2 млн + 1.5 млн = 3.5 млн
     Минус Retainage 10% = 350 тыс.
     К оплате: 3.15 млн
  → Ответственный менеджер проверяет объём
  → Директор утверждает
  → Казначей платит
  → Субподрядчик подписывает АВР
```

**Важно для налогов:** дата подписания АВР = дата признания расхода, независимо от даты оплаты.  
Стратегия: ускорить подписание АВР в конце квартала = уменьшить налог в этом периоде.

**Аванс** = payment against SOV line. Аванс не произвольный — он привязан к конкретному разделу работ.

### 2.4 Складской учёт (Warehouse)

```
Закупка материала → поступление на склад (GRN/Goods Receipt)
                              ↓
                    Материал = АКТИВ (не расход!)
                              ↓
                    Выдача в производство (Material Issue)
                              ↓
                    Материал = РАСХОД (Job Cost)
                              ↓
                    Остатки на складе = Инвентарь
```

Без склада: расход признаётся при закупке → завышение налогооблагаемой базы.  
Со складом: расход признаётся при выдаче → точный job costing.

### 2.5 Routing (маршрутизация)

Procore использует концепцию **«Ball in Court»** — у каждого элемента есть текущий владелец:

```
Создана PR → Ball: Approver
Approved  → Ball: Procurement (снабжение)
PO Created → Ball: Vendor
Delivered  → Ball: Site Manager (приёмщик)
GRN Signed → Ball: Accounting
Invoice OK → Ball: Treasury
Paid       → ✓ Closed
```

**Матрица ответственности для авто-роутинга:**

```
Правило 1: MATERIAL + объект → назначить снабженцу по объекту
Правило 2: SERVICE + объект + известный договор → прямо на утверждение
Правило 3: SERVICE + объект + нет договора → в пул (ручной выбор)
Правило 4: нет подходящего правила → в общий пул
```

---

### 2.6 Субподрядный лицевой счёт (Subcontractor Ledger)

Защита от проблемы: «Ернар должен нам деньги по объекту А, а Илья не знает и делает аванс по объекту Б».

```
Ернар Нурланов | Все проекты

Проект А (отв.: Максим):
  Договор:           10 000 000 ₸
  Выплачено авансов:  2 000 000 ₸
  АВР подписано:      1 500 000 ₸
  Долг Ернара:          500 000 ₸  ← аванс не покрыт работами

Проект Б (отв.: Илья):
  Договор:            5 000 000 ₸
  Выплачено авансов:          0 ₸
─────────────────────────────────────
ИТОГО долг Ернара:      500 000 ₸
```

Илья видит эту карточку перед созданием аванса → принимает информированное решение.

## 3. Практики для «плохих проектов» (неточное проектирование)

Распространённая ситуация на развивающихся рынках: формальный проект есть, но реальность расходится значительно, закрывается допсоглашениями постфактум.

### 3.1 Commitment-based controls (вместо budget-based)

Не контролировать через ненадёжный бюджет. Контролировать через обязательства:
> «Ничего не заказывается без PO или договора»

Работает даже при отсутствии точного бюджета.

### 3.2 Living Budget + Change Orders

```
Базовый бюджет (из сметы)     = 100 млн
+ Доп. соглашение №1           = +12 млн
+ Доп. соглашение №2           = +8 млн
──────────────────────────────────────────
Текущий бюджет                 = 120 млн  ← всегда актуален
```

Каждое допсоглашение = Change Order в системе. Бюджет всегда отражает договорную реальность.

### 3.3 Tolerance bands (пороги утверждения)

```
Потребность в рамках бюджетной строки    → авто (до 500к)
Превышение строки на < 20%               → утверждение руководителя
Превышение строки на > 20%               → требует Change Order
Потребность без бюджетной строки         → утверждение директора
```

### 3.4 «Мягкая» привязка к проекту

`project_id` = nullable в purchase_items. Позиция может существовать без проекта.  
Система показывает предупреждение — но не блокирует операции.

## 4. Маппинг на MidesCloud

| Индустриальный термин | MidesCloud сейчас | Нужно |
|---|---|---|
| **Project** | `objects` (частично) | Отдельная сущность `projects` |
| **WBS / Cost Code** | `work_types` (плоский) | Иерархический код с бюджетом |
| **Budget + Change Orders** | ❌ | `project_budget`, `change_orders` |
| **Purchase Requisition** | `purchase_items` (строится) | + project_id + cost_type |
| **Purchase Order** | ❌ | Между PR-approval и payment |
| **Goods Receipt / GRN** | ❌ | Приёмка материала |
| **Warehouse / Material Issue** | ❌ | Складской учёт |
| **Three-Way Match** | ❌ | Логика сопоставления |
| **Subcontract / Commitment** | `contracts` (частично) | + SOV (Schedule of Values) |
| **Payment Application** | `requests` ✓ | + привязка к SOV строке |
| **Retainage** | ❌ | Поле в контракте/платеже |

## 5. Приоритеты для MidesCloud

### Сделать немедленно (не блокирует, но предотвращает рефакторинг)

```sql
CREATE TABLE projects (
  id UUID PRIMARY KEY,
  name TEXT NOT NULL,
  object_id UUID REFERENCES objects(id),  -- первичный объект
  client_contract_value DECIMAL,
  budget_baseline DECIMAL,
  status TEXT DEFAULT 'ACTIVE',           -- BIDDING/ACTIVE/CLOSEOUT/CLOSED
  created_at TIMESTAMP
);

ALTER TABLE purchase_items
  ADD COLUMN project_id UUID REFERENCES projects(id),  -- nullable
  ADD COLUMN cost_type TEXT;  -- MATERIAL/LABOR/EQUIPMENT/SUBCONTRACT
```

### Следующая итерация

- `change_orders` (допсоглашения → обновление бюджета)
- `routing_rules` (матрица автоназначения)
- Субподрядный лицевой счёт (агрегат по contractor)

### Отложить

- Schedule of Values (SOV) — после стабилизации contracts
- Goods Receipt / Warehouse — после стабилизации закупок
- Three-Way Match — после PO
- Job Costing Report — после Cost Codes + Warehouse
