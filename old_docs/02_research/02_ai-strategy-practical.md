# AI-стратегия MidesCloud: практическое применение ИИ в аналоговой стройке

**Тип:** Explanation  
**Дата:** 2026-05-15  
**Контекст:** Где ИИ реально помогает строительной SMB-компании, а где остаётся игрушкой.  
**Фильтр:** Исключены задачи с высокой долей абстракции (стратегия, позиционирование, культурный контекст)

---

## Принцип отбора

Стройка — аналоговый бизнес. Проекты плохие, заказчики непрофессиональные,
половина расчётов идёт наличными, а данные живут в WhatsApp.

**ИИ полезен только там, где:**
1. Входные данные — документ, фото, число (не «ощущение» и не «отношения»)
2. Результат проверяем человеком за секунды
3. Ошибка ИИ не катастрофична (есть человек-валидатор)
4. Ручная альтернатива отнимает > 15 минут регулярно

---

## Карта применений по уровням процесса

### Уровень 1: ДОКУМЕНТЫ — распознавание и классификация

**Уже работает:**

| Задача | Вход | ИИ делает | Человек проверяет |
|--------|------|-----------|-------------------|
| Распознавание счетов | Скан/фото PDF | Извлечение: поставщик, БИН, позиции, суммы | Снабженец — 30 сек |
| Распознавание КС-2 | PDF | Строки: работа, ед.изм., объём, цена | ПТО — 2 мин |

**Следующие шаги:**

| Задача | Вход | ИИ делает | Ценность |
|--------|------|-----------|----------|
| Распознавание АВР | Скан АВР субподрядчика | Извлечение строк + автосверка с contract_items | Бухгалтер экономит 1-2 часа на АВР |
| Классификация входящих документов | Файл в Storage | Определение типа: счёт / АВР / КП / акт сверки / ЭСФ | Автоматическая маршрутизация |
| Извлечение из ЭСФ | XML/PDF | Сверка с purchase_items: суммы, реквизиты, НДС | Бухгалтер видит расхождения сразу |

### Уровень 2: МАППИНГ И СВЕРКА — сопоставление данных

**Это ваш следующий приоритет (уже запланирован):**

| Задача | Вход | ИИ делает | Без ИИ |
|--------|------|-----------|--------|
| Маппинг счёт → purchase_items | Распознанный счёт + реестр потребностей | Предлагает соответствие позиций (нечёткий поиск) | Снабженец вручную ищет 5-20 мин |
| Маппинг АВР → ВОР (contract_items) | Строки АВР + ВОР заказчика | Автосопоставление по описанию + ед.изм. + cost_code | ПТО тратит 30-60 мин на маппинг |
| Сверка КС-2 ↔ физическая приёмка | Данные из daily_logs + АВР | Флаг: «объём в АВР > объём в daily logs» | Сейчас никто не проверяет |
| Сверка КП поставщиков | 3-5 распознанных КП | Таблица сравнения + подсветка аномалий | Снабженец делает вручную в Excel |

**Ключевое:** маппинг — это не binary match, а ranked suggestions. ИИ предлагает,
человек подтверждает одним кликом. Если первый вариант верный в 80% случаев —
это уже экономия тысяч часов в год.

### Уровень 3: АНОМАЛИИ И КОНТРОЛЬ — защита от ошибок и потерь

Здесь ИИ работает как «второй глаз», который никогда не устаёт.

| Задача | Данные | ИИ делает | Реальная проблема |
|--------|--------|-----------|-------------------|
| Выброс по цене | История закупок + новый счёт | «Цемент M500: 4200 тг/т, средняя за 6 мес: 3100 тг/т» | Снабженец не помнит все цены |
| Дубли оплат | payments + purchase_items | «Похожая оплата тому же контрагенту 3 дня назад» | Казначей платит по реестру, не видит дубль |
| Превышение бюджета по cost_code | SUM(purchase_items) vs budget | «Бетонные работы: 112% бюджета, осталось 40% объёма» | Директор узнаёт слишком поздно |
| Субподрядчик «пухнет» | АВР: qty_claimed > qty_in_contract | «Объём арматуры в АВР на 18% больше ВОР» | ПТО ловит это, но не всегда |
| Просроченные документы | doc_collection_tasks.sla_deadline | «5 позиций без закрывающих документов > 14 дней» | Бухгалтер напоминает вручную |
| Истечение лицензии | contractor_compliance.valid_until | Автоуведомление за 30 дней | Сейчас бухгалтер «помнит» |

### Уровень 4: DAILY LOGS — превращение хаоса WhatsApp в данные

Это самое ценное применение ИИ для МиДЭС прямо сейчас.

**Текущий процесс:**
```
Прораб пишет в WhatsApp: "Сегодня залили 12 кубов бетона блок Б,
кран работал 4 часа, Серик не вышел, дождь до 11:00"

Офисный сотрудник → переносит в Excel → данные теряются
```

**С ИИ:**
```
1. Прораб пишет/диктует голосом в систему (или в WhatsApp с ботом)
2. ИИ парсит:
   - Работы: бетон, 12 м³, блок Б
   - Техника: кран, 4 часа
   - Персонал: Серик — отсутствие
   - Погода: дождь до 11:00
   - Простой: до 11:00, причина — погода
3. Система создаёт структурированную запись daily_log
4. Прораб подтверждает одним тапом
```

**Почему это реалистично:**
- Прораб НЕ будет заполнять форму с 20 полями
- Прораб БУДЕТ диктовать голосом или писать текстом
- Gemini отлично парсит неструктурированный русский текст в JSON
- Проверка человеком = 10 секунд

### Уровень 5: ГЕНЕРАЦИЯ ДОКУМЕНТОВ — ИИ собирает, человек подписывает

| Документ | Данные из системы | ИИ делает |
|----------|------------------|-----------|
| АВР для субподрядчика | contract_items + physical_acceptance + daily_logs | Генерирует форму Р1 / КС-2 с правильными объёмами |
| КС-2 заказчику | progress_claims + contract_items | Формирует документ по шаблону с накоплением |
| Акт сверки | payments + invoices по контрагенту | Готовый документ для подписи |
| Справка о стоимости (КС-3) | Агрегация по КС-2 за период | Автосводка |
| Письмо-запрос КП | purchase_items (описание, qty, unit) | Текст запроса + список поставщиков из истории |

**Ключевое:** субподрядчик приносит свою АВР с ошибками → система знает правильные
объёмы из daily_logs и contract_items → генерирует корректную АВР → субподрядчик
получает готовый документ на подпись. Меньше итераций, меньше ошибок.

### Уровень 6: ПРОГНОЗИРОВАНИЕ — когда данных станет достаточно (M4+)

Эти применения станут возможны когда будет 6-12 месяцев данных в системе:

| Задача | Данные | Результат |
|--------|--------|-----------|
| Прогноз cash flow | payment_tasks + scheduling | «Через 2 недели потребуется 15M тг на бетон» |
| Прогноз задержки | daily_logs + план | «Блок Б: при текущем темпе +12 дней к сроку» |
| Рейтинг поставщиков | sourcing_results + quality data | «Поставщик X: цена -5%, но срок доставки +3 дня» |
| Оптимальное распределение снабженцев | sourcing_tasks.completion_time | «Снабженец А быстрее на стройматериалах, Б — на МЕП» |

## Что НЕ стоит автоматизировать ИИ

| Область | Почему нет |
|---------|-----------|
| Выбор субподрядчика | Решение = отношения + репутация + личный опыт |
| Утверждение платежей | Юридическая ответственность — только человек |
| Переговоры с заказчиком | Культурный контекст, личные договорённости |
| Оценка качества работ | Требует физического присутствия |
| Финальное ценообразование | Слишком много неформальных факторов |

## Рекомендуемый порядок внедрения

```
СЕЙЧАС (M1):
  ✅ OCR счетов (работает)
  → Добавить: классификация входящих документов
  → Добавить: OCR АВР субподрядчика

M2 (Field Operations):
  → Daily Logs: текст/голос → структурированные данные
  → Аномалии: выбросы по ценам, дубли оплат

M3 (Revenue):
  → Маппинг АВР → ВОР (автосопоставление)
  → Генерация КС-2/КС-3 из данных системы
  → Сверка: объём в АВР vs объём в daily_logs

M4 (Financial Control):
  → Превышение бюджета по cost_code — автоалерт
  → Прогноз cash flow (когда будут данные за 6+ мес)

M6 (Subcontractor Portal):
  → Генерация АВР для субподрядчика
  → Автосверка входящей АВР с данными системы
```

## Архитектурное правило

> **Все AI-вызовы = Edge Functions в Supabase.**
> Модель: Gemini (flash-lite для OCR, flash для маппинга, pro для сложных сверок).
> Результат ИИ ВСЕГДА = DRAFT. Человек подтверждает → SUBMITTED → COMMITTED.
> ИИ НЕ ПИШЕТ в таблицы напрямую. Только через advance_workflow().

Это правило согласовано с ADR-38 (Task-per-Role) и инвариантом #22 (один оркестратор).
