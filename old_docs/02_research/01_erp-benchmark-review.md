Ни одна из мировых систем не заточена под этот профиль:
- Procore/CMiC: слишком дорого и сложно для SMB в Казахстане
- Buildertrend: жилое строительство, нет КС-2/КС-3, нет 1С
- Jonas: HVAC/электрика, не генподряд
- Sage: legacy, нет мобильной полевой работы

**Вывод:** MidesCloud занимает пустую нишу — **cloud-native ERP для SMB генподрядчика в СНГ**.

## 3. Оценка архитектурных решений

### 3.1 Что MidesCloud делает ЛУЧШЕ лидеров

| Решение | Почему лучше | Кто из лидеров так не делает |
|---------|-------------|------------------------------|
| **Плюс-минус вместо Change Orders** | Нет бюрократии на каждое отклонение. Допсоглашение только при NET > порога | Sage/CMiC требуют формальный CO на каждое изменение |
| **Task-per-Role (ADR-38)** | Изоляция модулей, устойчивость к потере контекста, AI-friendly | CMiC = monolith, Sage = legacy modules |
| **OCR + Discrepancy Engine** | Gemini API для распознавания КС-2, автосверка с ВОР | Только CMiC NEXUS (2025) имеет сопоставимое |
| **Multi-Entity для SMB** | ТОО + ИП как единый контур с налоговым учётом на переводы | Procore не поддерживает multi-entity вообще |
| **AI-native architecture** | Документация = контекст для AI-агентов разработки | Ни один конкурент не проектирует систему для AI-разработки |

### 3.2 Что MidesCloud делает НА УРОВНЕ

| Решение | Стандарт отрасли | MidesCloud |
|---------|-----------------|------------|
| Financial Spine (cost_code мост) | ✅ Procore, CMiC | ✅ cost_code × project |
| Approval workflows | ✅ все | ✅ с делегированием |
| Physical acceptance before payment | ✅ CMiC, Sage | ✅ инвариант #3 |
| Retention tracking | ✅ все | 🟡 поле есть, логика M3 |
| Document lifecycle | ✅ Sage (versions) | ✅ VOIDED + COMMITTED |

### 3.3 Что MidesCloud делает ХУЖЕ или НЕ ДЕЛАЕТ

Ранжировано по критичности для МиДЭС:

#### 🔴 Критичные пробелы (влияют на бизнес сейчас)

**GAP-B: Ежедневные отчёты с площадки (Daily Logs)**

Каждый объект = прораб, который ежедневно должен фиксировать:
- Погода, условия работы
- Присутствие рабочих (свои + субподряд)
- Выполненные работы за день
- Происшествия, простои, причины

> **Почему критично:** Сейчас это WhatsApp → офисный сотрудник переносит в Excel.
> Данные теряются. При споре с субподрядчиком нет доказательной базы.
> Procore считает Daily Logs одним из ТОП-3 модулей.
> **Приоритет: ПЕРВАЯ ОЧЕРЕДЬ (M2)**

**GAP-E: Учёт ресурсов (люди + техника)**

На площадке работают:
- Собственные рабочие (табель)
- Арендованная техника (часы работы)
- Субподрядные бригады (факт присутствия)

> **Почему критично:** Без табеля невозможно корректно рассчитать себестоимость.
> Без учёта техники — невозможно контролировать расходы на аренду.
> Все лидеры (Procore, CMiC, Jonas) имеют это.

#### 🟡 Важные пробелы (влияют при масштабировании)

**GAP-A: Document Control (RFI, Submittals)**

Запросы информации (RFI) к проектировщику, передача документации — сейчас
через email и WhatsApp. В системе нет:
- Реестра RFI с дедлайнами
- Цепочки submittals (проектировщик → подрядчик → заказчик)
- Версионности чертежей

> **Оценка:** Не критично для 3-5 проектов. Станет критичным при 10+.
> **Комментарий директора:** Часто часть работ сделана по одной версии чертежей,
> а утверждена другая. Но плохие проекты дают большие зазоры для компенсации таких ошибок.
> Это не является настоящей болью.

**GAP-D: Календарное планирование (Scheduling)**

Gantt-диаграммы, зависимости между работами, критический путь.
Сейчас — в MS Project или Excel.

> **Оценка:** Можно жить с внешним инструментом. Но интеграция с cost_code
> дала бы возможность прогнозировать cash flow (когда какие расходы).

**GAP-F: Мобильное приложение**

Полевые сотрудники (прорабы, ПТО) работают с телефона.
PWA или native app для:
- Daily logs
- Фотоотчёты с привязкой к объекту
- Приёмка работ
- Просмотр чертежей

> **Оценка:** Критичность растёт с каждым новым объектом.

#### ⚪ Некритичные пробелы (M5+)

- **GAP-C:** Safety module (инспекции, инциденты) — нормативно не требуется для SMB в КЗ
- **Client Portal** — полезно, но не блокирует операции
- **Preconstruction (bidding, estimating)** — пока не масштаб

## 5. Утверждённый roadmap (2026-05-15)

```
ПЕРВАЯ ОЧЕРЕДЬ:
  M1: P2P рефакторинг (ADR-38, task-per-role)          ← текущий
  M2: Field Operations
      ├── Daily Logs (GAP-B) — WhatsApp → система
      ├── Resource Tracking (GAP-E) — табель + техника
      └── Contractor compliance автоуведомления
  M3: Revenue (client billing, КС-2/3, маржа)

ВТОРАЯ ОЧЕРЕДЬ:
  M4: Financial Control (budget vs fact, WIP, bank reconciliation)
  M5: Scheduling + Cash Flow Forecasting (GAP-D)

ТРЕТЬЯ ОЧЕРЕДЬ:
  M6: Mobile App (GAP-F) + Портал субподрядчика
      └── Субподрядчик подаёт запрос на приёмку →
          утверждения → система формирует АВР →
          печать или подпись ЭЦП
  M7: Складской учёт
  M8: HR
```

**Обоснование:** Procore ($15B компания) считает Daily Logs и Field Reports
**основой платформы**, а не финансы. Данные с площадки → решения в офисе.
Без полевых данных финансовый контроль (M4) будет неполным.

## 6. Итоговая оценка

### Архитектура: 9/10

Blueprint v2.1 архитектурно превосходит то, что предлагают Sage 300 и Buildertrend:
- Task-per-Role = лучшая изоляция модулей, чем monolith CMiC
- Computed views = меньше рассинхрона, чем stored status у Sage
- AI-native документация = уникальное преимущество

Единственный минус: архитектура Revenue side (M3) пока только намечена.

### Функционал: 4/10

По объёму функций MidesCloud покрывает ~30% того, что даёт Procore.
Но это нормально для стадии M1. Критичные пробелы (Daily Logs, Resource Tracking)
должны быть закрыты в M2, иначе система останется «только для офиса».

### Уникальные преимущества: 3 штуки

1. **Multi-Entity для SMB** — ни Procore, ни Buildertrend этого не делают
2. **КС-2/КС-3/Р1 нативно** — западные системы не знают этих форматов
3. **AI-powered OCR** — на уровне CMiC NEXUS, но для SMB
