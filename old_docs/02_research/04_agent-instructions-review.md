## Назначение документа

Настоящий документ представляет собой детальный критический анализ и ревью инструкций, предназначенных для автономных AI-агентов в рамках проекта MidesCloud v3. Анализ сфокусирован на следующих файлах:
1. `.antigravityrules` — глобальные правила проекта и запрещенные паттерны.
2. `AGENTS.md` — контракты агентов, pre-read цепочки и ограничения (Guard Rails).
3. `docs/00_index/02_agent-routing.md` — матрица маршрутизации задач.
4. `docs/08_reference/02_development-skills-guide.md` — распределение AI-скилов по фазам.
5. `web/scripts/context-health.ts` — автоматический детектор дрейфа контекста (enforcement).

## 1. Сильные стороны (Что делает среду «Agent-First»)

Текущая конфигурация представляет собой одну из лучших практик для автономной AI-разработки. Ключевые преимущества:

*   **Локализация контекста (Pre-read Pipelines):** AI-агенты крайне чувствительны к объему контекста. Четкие цепочки файлов для обязательного прочтения (Mandatory Pre-read) позволяют агенту сфокусироваться на нужных правилах и доменной модели, не сканируя весь проект целиком.
*   **Автоматическое предотвращение дрейфа (Drift Detection):** Наличие скрипта `context-health.ts`, запускаемого в CI, предотвращает классическую проблему AI-разработки — «уплывание» документации и кодовой базы друг от друга. Он проверяет синхронизацию токенов CSS, экспорт схем, отсутствие мусора (`$path`) и неверную длину компонентов.
*   **Защитные барьеры (Guard Rails):** Ограничение длины UI-компонентов до 150 строк заставляет агентов разбивать код на переиспользуемые атомарные части, предотвращая генерацию гигантских спагетти-файлов. Запреты на модальные окна (только листы/sheets) и запрет Geist/Arial предотвращают визуальный хаос.
*   **Принципы Gall's Law & Vertical Slicing:** Четко прописанный запрет на «Big Bang» разработку (сначала вся БД, потом весь бэк, потом UI) заставляет агентов внедрять фичи вертикальными срезами (DB → API → UI), что минимизирует риски ошибок проектирования.

## 2. Обнаруженные противоречия и дрейф контекста

В ходе глубокого ревью были выявлены следующие расхождения между декларативными инструкциями и технической реализацией:

### 2.1 Противоречие блокировки CI при предупреждениях (Warnings)
*   **Декларация:** В `docs/01_product/02_project-goals.md` (раздел Goals A1-A5) и в `.antigravityrules` §9 указано: *"CI blocks on warnings"* (CI блокирует сборку при наличии предупреждений).
*   **Реальность:** В скрипте `web/scripts/context-health.ts` (строки 350-352) логика обработки предупреждений выглядит так:
    ```typescript
    } else if (warnCount > 0) {
      console.log("⚠️  Context health check PASSED WITH WARNINGS — review items above.\n");
      process.exit(0);
    }
    ```
    Возврат кода `0` означает, что CI-сервер (GitHub Actions) считает шаг успешно пройденным и **не блокирует** сборку.
*   **Анализ:** Exiting с `0` при предупреждениях — это верное инженерное решение, иначе мелкие предупреждения (например, временный выход компонента за лимит 150 строк на этапе черновика) парализовали бы CI. Однако формулировка в `02_project-goals.md` должна быть скорректирована для соответствия реальности.

### 2.2 Устаревшие ссылки в Pre-read цепочках
*   **В `AGENTS.md` §3 (Backend & Security Agent):**
    В списке pre-read указан файл `docs/03_decisions/36_unified-payment-registry.md`. Однако этот ADR имеет статус **Partially Superseded** (частично заменен) решением `docs/03_decisions/38_task-per-role-architecture.md` (архитектура тонкого якоря и задач ролей).
    *Рекомендация:* Заменить или дополнить эту ссылку на ADR-38, так как чтение устаревшей версии 36-го решения может направить Backend-агента по ложному пути.
*   **В `AGENTS.md` §1 (Product Agent):**
    Указан пункт *"Relevant ADRs from docs/03_decisions/ (check index for topic)"*. Это слишком размытая инструкция для AI.
    *Рекомендация:* Указать, что Product-агент должен начинать с `docs/00_index/01_docs-index.md` для поиска пересечений, чтобы не плодить дубликаты ADR.

### 2.3 «Слепые зоны» автоматической проверки (Guard Rails без контроля)
Ряд жестких запретов из `AGENTS.md` и `.antigravityrules` никак не проверяются скриптом `context-health.ts`, что оставляет пространство для ошибок:
1.  **Escape-to-close для Sheet:** Frontend-агент обязан оборачивать все Sheet в хук `useSheetEsc(open, onClose)`. Скрипт автопроверки это не проверяет.
2.  **Запрет жесткого кодирования ролей:** Запрещены *"Free-text Role Strings"*, но нет автоматической проверки на использование сырых строковых литералов (например, `"DIRECTOR"` вместо константы/перечисления) в коде Server Actions или страниц.
3.  **Использование форматирования валюты:** Правило *"ALL monetary amounts → formatCurrency"* не проверяется статическим анализатором.

## 3. Предлагаемый план устранения замечаний

Для повышения качества автономной работы агентов предлагаются следующие доработки:

### Шаг 1: Обновление декларативной документации
1.  В `docs/01_product/02_project-goals.md` исправить формулировку: указать, что CI блокирует сборку при критических ошибках (Failures), а предупреждения (Warnings) выводятся в консоль для ручного контроля, но не останавливают пайплайн.
2.  В `AGENTS.md` §3 обновить обязательный pre-read: заменить `36_unified-payment-registry.md` на актуальный `38_task-per-role-architecture.md`.

### Шаг 2: Усиление автоматического контроля (Lint & Context Check)
Добавить в `web/scripts/context-health.ts` простые статические проверки:
1.  **Проверка `useSheetEsc`:** Сканировать файлы компонентов, импортирующих `SheetContent` из `@/components/ui/sheet`, и проверять наличие вызова `useSheetEsc`.
2.  **Проверка сырых ролей:** Искать использование строковых литералов ролей в нетипизированных местах.
