# MidesCloud — Карта конвееров (Value Stream Map)

**Статус:** УТВЕРЖДЕНО (v3)  
**Дата:** 2026-05-18  
**Основа:** SAP PS + Procore + CMiC + Lean Construction  
**Формат:** L1 → L2 → L3

---

## L1: Главный конвеер — «Тендер → Деньги» (Bid-to-Cash)

> Частица = **Проект**. Этапы 2→3→4 цикличны.

```
[0] Коммерческая    [1] Запуск       [2] Производство    [3] Сдача работ    [4] Получение    [5] Закрытие
    подготовка          проекта          работ                заказчику          оплаты           проекта
─────────────► ──────────► ──────────────► ───────────────► ────────────► ──────────►
```

| # | Этап | Вход → Выход | Владелец |
|---|------|-------------|----------|
| 0 | Коммерческая подготовка | Тендер → Подписанный договор + ВОР | Директор + команда оценки |
| 1 | Запуск проекта | Договор → Проект + субподрядные договоры | PM + GI |
| 2 | Производство работ | Проект → Выполненные + принятые работы | Начальник участка |
| 3 | Сдача работ заказчику | Принятые работы → Подписанные КС-2/КС-3 + счёт | ПТО + PM |
| 4 | Получение оплаты | Счёт → Деньги на счёте | Бухгалтер |
| 5 | Закрытие проекта | Оплаченный проект → Закрытый проект | PM + Экономист |

---

## Карта вложенных конвееров

```
L1: Тендер → Деньги
│
├─ [0] Коммерческая подготовка
│   └── L2-PRE: Предстроительный конвеер
│
├─ [1] Запуск проекта
│   └── L2-CON: Контрактование субподрядчиков
│
├─ [2] Производство работ
│   ├── L2-A: Снабжение участка (материалы, услуги, техника)
│   │   ├── L3-A1: Sourcing
│   │   └── L3-A2: Проверка остатков (будущее)
│   │
│   ├── L2-B: Субподрядные работы (цикл приёмок)
│   │   └── L3-B1: Discrepancy Review
│   │
│   ├── L2-FIELD: Полевая отчётность и прогресс
│   │
│   └── L2-HR: Табелирование и ЗП
│
├─ [3] Сдача работ заказчику
│   └── L2-D: Биллинг заказчика
│       └── L3-D1: Плюс-минус (вариации)
│
├─ [4-5] Получение оплаты / Закрытие
│
├─ [Сквозной] L2-FIN: Финансовый контроль
│
└─ [Сквозной] L2-ACC: Бухгалтерский конвеер (в 1С)
```

**Ретро** = альтернативный выход в L2-A / L2-B / L2-HR. Тот же маршрут, постфактум.

---

## L2-PRE: Предстроительный конвеер

> **Триггер:** Тендер / запрос заказчика  
> **Владелец:** Директор

| Шаг | Кто | Что | Gate |
|-----|-----|-----|------|
| PRE.1 | Директор | Go / no-go: ресурсы, сроки, риски | **Gate: решение участвовать** |
| PRE.2 | Директор | Собирает команду оценки (временная роль «экономист») | — |
| PRE.3 | Команда оценки | Расчёт себестоимости: материалы, субподряд, техника, ФОТ, накладные. Источник: `price_history` | — |
| PRE.4 | Команда + Директор | Себестоимость + маржа → конкурентное КП | Gate: КП утверждено |
| PRE.5 | Директор | Подача, переговоры | — |
| PRE.6 | Директор + Юрист (проверка) | Подписание договора, фиксация ВОР | **Gate: договор подписан** |

> Кто-то из команды оценки с высокой вероятностью становится PM проекта.

---

## L2-CON: Контрактование субподрядчиков

> **Триггер:** ВОР заказчика декомпозирован → нужны субподрядчики  
> **Владелец:** PM (все переговоры ведёт PM, юрист только проверяет документы)

| Шаг | Кто | Что | Gate |
|-----|-----|-----|------|
| CON.1 | GI + ПТО | ВОР заказчика → пакеты по видам работ | — |
| CON.2 | PM | Поиск субчиков, запрос КП, проверка compliance | — |
| CON.3 | PM + Директор | Выбор, согласование цены | **Gate: Директор утвердил** |
| CON.4 | Юрист | Проверка документов договора | Gate: документы ОК |
| CON.5 | PM | ВОР субчика (из ВОР заказчика), подписание | **Gate: договор подписан** |

> **Ключевое:** `cost_code_id` связывает ВОР заказчика → ВОР субчика → purchase_items.

---

## L2-A: Снабжение участка

> Материалы, услуги, аренда техники — один конвеер, разные документы на выходе.

| Шаг | Кто | Что | Gate |
|-----|-----|-----|------|
| A1 | Начальник участка / Прораб | Заявка: что, сколько, когда, объект. Сейчас = всегда покупка (нет видимости остатков) | — |
| A2 | Начальник снабжения | Распределяет позиции между снабженцами | — |
| A3 | Снабженец | **→ L3-A1 (Sourcing)** | Gate: оплата подтверждена |
| A4 | Снабженец | Контроль доставки на объект / приезд техники | — |
| A5 | Кладовщик | Приёмка на склад, проверка кол-ва и качества (GRN) | **Gate: Three-Way Match** |
| A6 | Прораб | Выдача в работу под роспись (списание со склада) | — |
| A7 | Снабженец → Бухгалтер | Первичные документы → ЭСФ → бухгалтерия | Gate: передано |

### L3-A1: Sourcing

```
Запрос КП (мин. 3) → Получение счетов → Сравнение → Выбор поставщика →
→ Утверждение (GI→PM→Директор) → Оплата казначеем
```

### L3-A2: Проверка остатков (будущее)

> Между A1 и A2: система проверяет все склады. Если есть → внутреннее перемещение, не закупка.

## L2-B: Субподрядные работы (цикл приёмок)

> **Частица:** Одна приёмка работ (событийный).  
> **Цикл:** Субчик сделал → принял → оплатил → следующий объём.

### Модель авансов (Subcontractor Ledger)

> Баланс = SUM(все оплаты) − SUM(COMMITTED документы).  
> Баланс > 0 → субчик должен компании. Баланс < 0 → компания должна субчику.

**Путь 1: Стандартный** (работы → документы → оплата)

| Шаг | Кто | Gate |
|-----|-----|------|
| B1 | Субподрядчик | Выполнил объём, готов к приёмке |
| B2 | SITE_MANAGER | Физическая приёмка | **Gate: ACCEPTED / PARTIAL / REJECTED** |
| B3 | PM / Субподрядчик | Документы + OCR → L3-B1 Discrepancy Review | Gate: данные извлечены |
| B4 | ПТО + Бухгалтер | Проверка параллельно | **Gate: оба одобрили → COMMITTED** |
| B5 | GI → PM → Директор | Утверждение | **Gate: Директор утвердил** |
| B6 | Казначей | Оплата субчику | Gate: оплачено |
| B7 | PM → Бухгалтер | Закрывающие документы → бухгалтерия | Gate: передано |

**Путь 2: Аванс** (оплата → потом документы закрывают долг)

| Шаг | Кто | Gate |
|-----|-----|------|
| B-ADV.1 | PM | Запрос аванса | — |
| B-ADV.2 | Директор | Утверждение аванса | **Gate: утверждён** |
| B-ADV.3 | Казначей | Оплата аванса (долг субчика вырос) | Gate: оплачено |

> Когда работы выполнены: B1 → B2 → B3 → B4 → **пропуск B5/B6** → B7.  
> Документы закрывают существующий долг. Повторное утверждение **не нужно**.

### Обработка ошибок

- B2: SITE_MANAGER не принял → конвеер НЕ запускается
- B3/B4: Ошибка → итерация АННУЛИРУЕТСЯ + уведомление всех участников. Документы переделываются с нуля.

## L2-FIELD: Полевая отчётность и прогресс

> **Триггер:** Наличие ГПР по проекту  
> **Владелец:** Прораб → PM

| Шаг | Кто | Что |
|-----|-----|-----|
| FIELD.1 | PM / GI + Система | **Система помогает создать ГПР**: каждой работе — время начала/конца, связи между работами (нельзя начать без завершения предыдущей) |
| FIELD.2 | Система → Прораб | ГПР даёт прорабу информацию: что нужно делать |
| FIELD.3 | Прораб | Периодический отчёт: что сделано, объёмы, фото |
| FIELD.4 | Система | Агрегация отчётов в накопитель прогресса по cost_code |
| FIELD.5 | PM | Сравнение факта с планом ГПР → корректировка плана |

> **Связь с L2-D:** Накопитель прогресса = основа для КС-2.  
> **Связь с L2-B:** Физическая приёмка (B2) подтверждает данные отчётности.

---

## L2-HR: Табелирование и ЗП

| Шаг | Кто | Что | Gate |
|-----|-----|-----|------|
| HR.1 | Прораб / Нач. участка | Ежедневный учёт: кто, где, сколько часов | — |
| HR.2 | Система | Сводка за период по сотруднику × проект × cost_code | — |
| HR.3 | Нач. участка → Директор | Утверждение табеля | **Gate: утверждён** |
| HR.4 | Бухгалтер / HR | Расчёт: ставка × часы + надбавки − вычеты | — |
| HR.5 | Казначей | Выплата ЗП | Gate: выплачено |

> **Сезонные рабочие** (не наняты официально, расчёт наличными) = **ретро-операция**.  
> Часы × ставка → себестоимость труда по cost_code.

---

## L2-D: Биллинг заказчика (Revenue)

| Шаг | Кто | Что | Gate |
|-----|-----|-----|------|
| D1 | ПТО | Маппинг принятых АВР на строки ВОР заказчика | — |
| D2 | ПТО | Формирование КС-2 (объёмы) + КС-3 (стоимость) | Gate: суммы сходятся |
| D3 | PM + Заказчик | Подписание актов | **Gate: заказчик подписал** |
| D4 | Бухгалтер | Выставление счёта | — |
| D5 | Бухгалтер / Экономист | Контроль оплаты | Gate: оплата получена |

> **L3-D1: Плюс-минус** — NET = SUM(OVERRUN) − SUM(UNDERRUN). NET > порога → допсоглашение.

---

## L2-ACC: Бухгалтерский конвеер (в 1С)

> Полноценный конвеер, не точка сбора. MidesCloud поставляет данные → получает обратно список ЭСФ.

| Шаг | Кто | Что | Gate |
|-----|-----|-----|------|
| ACC.1 | Система → Бухгалтер | Документы из L2-A, L2-B, L2-HR, L2-D | — |
| ACC.2 | Бухгалтер | Проверка бумажных документов: реквизиты, подписи | Gate: корректны |
| ACC.3 | Бухгалтер | Сверка бумажного документа с ЭСФ | **Gate: совпадает** |
| ACC.4 | Бухгалтер | Проводка в базу 1С | Gate: проведено |
| ACC.5 | Бухгалтер | Архивирование бумажных документов | — |

> **Обратная связь:** MidesCloud получает от бухгалтерии **список выставленных ЭСФ** — и всё.  
> При несовпадении → уведомление PM/снабженцу → повторный сбор документов.

---

## L2-FIN: Финансовый контроль (сквозной)

| Метрика | Формула | Кому |
|---------|---------|------|
| Budget vs Actual | Бюджет − SUM(purchase_items) по cost_code | PM, Директор |
| Маржа проекта | SUM(contract_items) − SUM(purchase_items) | Экономист |
| Баланс контрагента | SUM(payments) − SUM(accepted_docs) | Казначей |
| Себестоимость труда | SUM(часы × ставка) по cost_code × проект | Экономист |
| WIP Report | Консолидированный по legal_entities | Директор |

> **Нет «реестра казначея».** Оплата = шаг в конкретном конвеере.

---

## Альтернативный выход: Ретро-операции

Не отдельный конвеер. Альтернативный **выход** из L2-A / L2-B / L2-HR.  
Тот же маршрут утверждения, постфактум. `is_retrospective = true` + причина.  
Badge «Ретро» в UI. Система не блокирует работу — фиксирует реальность.

Примеры: прораб заплатил наличными крановщику, сезонный рабочий без оформления.
