# Research: Анализ производительности реестра заявок и расчетных балансов (1000+ строк)

**Статус:** На обсуждении  
**Дата:** 2026-05-21  
**Связан с:** ADR-07 (accordion table), ADR-36 (unified-payment-registry), ADR-38 (task-per-role), ADR-42 (requests registry), ADR-46 (contractor-level-requests-nesting)

---

## 1. Введение

Данный документ представляет собой анализ потенциальных проблем масштабируемости текущей архитектуры реестра заявок (`/requests`, `/requests/approvals`, `/payments`) при накоплении значительного объема данных (более 1000 строк) на реальных строительных объектах, а также предлагает конкретные решения по оптимизации.

---

## 2. Анализ текущего подхода

### Проблема 1: Отсутствие пагинации и лимитов (Загрузка 1000+ строк)

#### Текущая реализация
- Метод `getCommitmentsForAll()` в `actions.ts` запрашивает все записи из таблицы `purchase_items` без ограничений (`LIMIT`/`OFFSET` или курсоры):
  ```typescript
  const { data: items, error } = await supabase
    .from("purchase_items")
    .select(`...`)
    .order("created_at", { ascending: false });
  ```
- Полученный массив целиком передается клиенту.
- Компонент `RequestsAccordionTree` на клиенте через `useMemo` производит группировку: Проект → Объект → Контрагент → Заявки.
- При рендере все узлы дерева раскрыты по умолчанию (`defaultOpen`).

#### Последствия на реальных данных
1. **Сетевой оверхед (Payload Bloat):** Каждая строка `purchase_items` содержит вложенные объекты (`project`, `object`, `contractor`, `commitmentApprovalTask`, `paymentTask`). При объеме 1000+ строк размер JSON-ответа составит 2–5 МБ, а при 10 000+ строках — превысит 20–50 МБ. Это приведет к медленной загрузке страницы, высокому расходу трафика и риску обрыва соединений на мобильных устройствах.
2. **Блокировка UI-потока (CPU Bottleneck):** Группировка тысяч элементов в JavaScript на стороне клиента выполняется в основном потоке (main thread). Каждое изменение фильтра или статуса будет вызывать пересчет `useMemo`, блокируя интерфейс и вызывая визуальные микрофризы (лаги).
3. **Перегрузка DOM (DOM Size Explosion):** Поскольку все уровни аккордеона раскрыты по умолчанию, браузер вынужден рендерить тысячи DOM-элементов одновременно. Это приводит к падению производительности скроллинга (FPS drop), долгой перерисовке стилей (Recalculate Styles) и риску аварийного завершения вкладки браузера (Out of Memory).

### Проблема 2: Динамический расчет балансов на сервере

#### Текущая реализация
- Функция `injectContractorMetrics` собирает уникальные `projectIds` из загруженных заявок.
- Затем она выполняет два «плоских» запроса ко всей истории таблиц `avr_tasks` и `payment_tasks` для данных проектов:
  ```typescript
  // Запрос всех АВР по проектам
  const { data: avrs } = await supabase
    .from("avr_tasks")
    .select("avr_amount, purchase_items!inner(project_id, object_id, contractor_id)")
    .eq("status", "APPROVED")
    .in("purchase_items.project_id", projectIds);

  // Запрос всех платежей по проектам
  const { data: payments } = await supabase
    .from("payment_tasks")
    .select("amount_paid, purchase_items!inner(project_id, object_id, contractor_id)")
    .in("status", ["PAID", "PARTIAL"])
    .in("purchase_items.project_id", projectIds);
  ```
- После этого сервер в памяти (Node.js) группирует эти данные через `Map` и суммирует значения для инъекции в каждый элемент списка.

#### Последствия на реальных данных
1. **Зависимость скорости от истории транзакций:** Время выполнения этих запросов растет линейно ($O(N)$) в зависимости от общего объема накопленных АВР и платежей по проектам. Даже если на странице нужно отобразить всего 50 последних заявок, сервер все равно выкачает *всю* историю АВР и оплат за несколько лет по связанным проектам для расчета баланса контрагентов.
2. **Нагрузка на базу данных (CPU/RAM DB):** Операция `!inner` JOIN на уровне базы данных для фильтрации по `project_id` без материализации результатов будет выполнять дорогостоящие сканирования таблиц (Seq Scan / Index Scan с большими хэш-таблицами) при каждом обращении к реестру.
3. **Бесполезный расход памяти Node.js:** Перебор массивов из десятков тысяч исторических записей АВР и оплат в оперативной памяти сервера при каждом открытии реестра приведет к быстрому исчерпанию ресурсов сервера (Node.js Heap Memory OutOfMemory).

## 3. Предлагаемые решения и оптимизация

Мы разделяем оптимизацию на тактические (быстрые улучшения) и стратегические (долгосрочные архитектурные решения) изменения.

### Решение А: Оптимизация вычисления балансов (Бэкенд / БД)

#### Вариант 1. Таблица агрегатов с триггерами (Идеальное решение)
Создается выделенная таблица для хранения предрассчитанных финансовых показателей контрагента в разрезе проектов и объектов.

1. **Создание таблицы `contractor_project_balances`:**
   ```sql
   CREATE TABLE contractor_project_balances (
     project_id UUID NOT NULL REFERENCES projects(id),
     object_id UUID REFERENCES project_objects(id),
     contractor_id UUID NOT NULL REFERENCES contractors(id),
     avr_sum NUMERIC(18, 2) DEFAULT 0.00 NOT NULL,
     paid_sum NUMERIC(18, 2) DEFAULT 0.00 NOT NULL,
     balance NUMERIC(18, 2) GENERATED ALWAYS AS (avr_sum - paid_sum) STORED,
     updated_at TIMESTAMP DEFAULT NOW() NOT NULL,
     PRIMARY KEY (project_id, COALESCE(object_id, '00000000-0000-0000-0000-000000000000'::uuid), contractor_id)
   );
   ```
2. **Базовые триггеры на изменение данных:**
   - **Триггер на `avr_tasks`:** При переходе АВР в статус `APPROVED` (или изменении суммы) инкрементировать `avr_sum` для соответствующего контрагента.
   - **Триггер на `payment_tasks`:** При переходе платежа в статус `PAID`/`PARTIAL` (или изменении суммы) инкрементировать `paid_sum`.
3. **Эффект:** Чтение метрик превращается в быструю операцию $O(1)$ по первичному ключу или индексу. Никаких джойнов и агрегаций в памяти Node.js при чтении реестра.

#### Вариант 2. База данных VIEW (Динамическая агрегация средствами PostgreSQL)
Вместо переноса всей логики расчетов в Node.js, создается SQL View, который перекладывает агрегацию на СУБД.

```sql
CREATE VIEW contractor_balances_view AS
SELECT 
  pi.project_id,
  pi.object_id,
  pi.contractor_id,
  COALESCE(SUM(CASE WHEN avr.status = 'APPROVED' THEN avr.avr_amount ELSE 0 END), 0) AS avr_sum,
  COALESCE(SUM(CASE WHEN pay.status IN ('PAID', 'PARTIAL') THEN pay.amount_paid ELSE 0 END), 0) AS paid_sum,
  (COALESCE(SUM(CASE WHEN avr.status = 'APPROVED' THEN avr.avr_amount ELSE 0 END), 0) - 
   COALESCE(SUM(CASE WHEN pay.status IN ('PAID', 'PARTIAL') THEN pay.amount_paid ELSE 0 END), 0)) AS balance
FROM purchase_items pi
LEFT JOIN avr_tasks avr ON avr.purchase_item_id = pi.id
LEFT JOIN payment_tasks pay ON pay.purchase_item_id = pi.id
GROUP BY pi.project_id, pi.object_id, pi.contractor_id;
```
- **Эффект:** На стороне Node.js Drizzle делает простой запрос к View с фильтрацией по нужным `project_id`. База данных оптимизирует группировку благодаря внешним ключам и индексам.

### Решение Б: Оптимизация интерфейса и рендеринга (Фронтенд)

#### 1. Серверная пагинация (Server-Side Pagination)
- Разделить реестр на страницы (например, по 50 записей).
- Внедрить параметры `limit` и `offset` (или курсорную пагинацию по `created_at` / `id`) в серверные экшены:
  ```typescript
  export async function getCommitmentsForPage(page: number = 1, limit: number = 50) { ... }
  ```

#### 2. Ленивая загрузка дерева (Lazy Loading Tree)
Вместо загрузки всех 4 уровней иерархии сразу, загружать данные по требованию:
1. **Первоначальный рендер:** Загружаются только активные Проекты (Уровень 1).
2. **Развертывание Проекта:** Отправляется быстрый запрос на получение Объектов (Уровень 2) данного проекта.
3. **Развертывание Объекта:** Загружаются Контрагенты (Уровень 3) с их балансами из предрассчитанной таблицы агрегатов.
4. **Развертывание Контрагента:** Загружаются конкретные Заявки (Уровень 4) данного контрагента.
- **Эффект:** Стартовый размер Payload снижается с мегабайтов до нескольких килобайтов. Браузер рендерит минимальное число DOM-узлов.

#### 3. Виртуализация списка (DOM Virtualization)
Если бизнесу критически важно иметь возможность развернуть все дерево сразу (без постраничного деления):
- Использовать библиотеку виртуализации (например, `@tanstack/react-virtual`).
- Браузер будет рендерить только те строки, которые физически видны на экране (около 30-40 строк), динамически заменяя их при скролле.
- **Эффект:** DOM-дерево остается стабильным и легковесным независимо от того, загружено ли 1 000 или 10 000 заявок.

#### 4. Изменение состояния по умолчанию
- Убрать авто-раскрытие всех уровней (`defaultOpen`).
- По умолчанию загружать дерево в свернутом виде (или раскрывать только первый проект/объект), чтобы первый рендер происходил мгновенно.

## 4. Сравнительная таблица решений

| Решение | Сложность реализации | Эффект на производительность | Риски | Рекомендация |
|---|---|---|---|---|
| **Агрегированная таблица + триггеры** | Средняя | Максимальный (Бэкенд $O(1)$) | Необходимость синхронизации исторических данных при миграции | **Рекомендуется** как основное серверное решение |
| **SQL View** | Низкая | Средний (СУБД рассчитывает на лету) | При огромной истории может замедляться без материализации | Хорошая временная альтернатива |
| **Ленивая загрузка дерева** | Высокая | Высокий (Снижение Payload) | Усложнение стейт-менеджмента на фронтенде | **Рекомендуется** для UX |
| **Пагинация / Лимиты** | Низкая | Высокий | Затрудняет сквозной подсчет агрегатов по всем страницам на клиенте | Необходима в любом случае |
| **Виртуализация DOM** | Средняя | Высокий (Устраняет лаги рендера) | Возможны артефакты при быстрой прокрутке | **Рекомендуется** при отказе от пагинации |
