# Anti-Big Bang: Порядок реализации конвеерной архитектуры v3

**Статус:** ЕДИНСТВЕННЫЙ РОАДМАП ПРОЕКТА  
**Роль:** Уровень 2 документации (Blueprint → **Roadmap** → Phase Plan)  
**Принцип:** Gall's Law — каждый работающий сложный системный модуль вырос из работающего простого  
**Версия:** v3 — исправлена формула Ledger, маршрут утверждения, связь субподрядчик→проект + Pre-Mortem

---

## Ключевое архитектурное решение: Unified Commitment

SAP, Procore и CMiC подтверждают: закупки материалов (PO) и субподряд (Subcontract) — **варианты одного Commitment-конвеера**.

```
commitment_type = SUBCONTRACT    → Physical Acceptance + АВР + Ledger + Гарантийное удержание
commitment_type = PURCHASE_ORDER → GRN (склад) + Three-Way Match
commitment_type = SERVICE        → Акт услуг (аренда, охрана)
commitment_type = ADVANCE        → Аванс → долг → закрытие документами
```

Один движок `advance_workflow()`, разные gates по типу.

### ⚠️ Уточнение: Как субподрядчик связывается с проектом?

Не напрямую. Только через договор:

```
Contractor → Contract (intent / full) → Project / Object
```

**Маршрут:**
1. Директор принимает решение привлечь субподрядчика (устное намерение / переговоры)
2. В системе создаётся `contract` со статусом `INTENT` — субподрядчик привязывается к проекту
3. Юрист получает **задачу** (lawyer_task): «Подготовить договор № X для подписания директором»
4. Директор подписывает → статус меняется на `SIGNED` → конвеер L2-COMMIT открывается

**Вывод для Фазы 0:** Foundation должен включать базовую карточку договора (не только Contractor) — без неё невозможно создать ни одно Commitment.

```
contracts.warranty_percent   -- задаётся при подписании, влияет на Ledger
contracts.warranty_period_months
```

## Фаза 1: L2-COMMIT — Subcontract Slice (3 недели)

> **Цель:** Самый болезненный маршрут end-to-end. Субподряд = 70%+ расходов.

### Маршрут утверждения (исправлено)

```
[1] PM создаёт заявку на оплату → [2] Попадает в общий реестр Директора → [3] Директор утверждает
→ [4] Казначей видит одобренные → [5] Оплата → [6] Физ.приёмка (начальник участка)
→ [7] Документы + OCR → [8] Проверка ПТО → [9] data_status = COMMITTED → [10] Бухгалтерия
```

> **Нет GI в цепочке утверждения субподрядных оплат.** PM → Директор напрямую (общий реестр заявок на утверждение).

### Subcontractor Ledger (исправлено)

```
Баланс субподрядчика = SUM(оплаты) − SUM(гарантийные_удержания) − SUM(COMMITTED документы)

Где:
  гарантийное_удержание = оплата × contract.warranty_percent / 100
  Задаётся при подписании договора (warranty_percent)
  Накапливается по каждому платежу
  Возвращается отдельным request типа WARRANTY_RELEASE после истечения гарантийного срока
```

**Пример:**
```
Договор №15, warranty_percent = 5%
Выплачено: 10 000 000 ₸
Гарантийное удержание: 500 000 ₸ (по 5% с каждого платежа)
COMMITTED документы: 8 000 000 ₸
Чистый долг субподрядчику: 10 000 000 − 500 000 − 8 000 000 = 1 500 000 ₸ переплата
```

### Авансовый путь

```
Запрос аванса → Директор утверждает → Казначей платит → Долг растёт
Долг = SUM(авансы) − SUM(COMMITTED документы, закрывающие аванс)
```

# PRE-MORTEM: Где мы споткнёмся в первую очередь?

> **Метод:** Представляем, что план провалился. Что пошло не так?  
> Вероятность оценивается по опыту строительных ERP-внедрений.

---

## 🔴 Риск #1: Договор как предусловие (ВЕРОЯТНОСТЬ: ОЧЕНЬ ВЫСОКАЯ)

**Что происходит:**
Фаза 0 требует создания договора перед любым Commitment. На практике субподрядчики начинают работать до подписания договора — «договорились на словах». В системе нет договора → невозможно создать заявку → конвеер не запускается → директор говорит «система не работает» → отказ от внедрения.

**Признак провала:**
> «Мы не можем создать заявку на оплату, потому что договора ещё нет, но субчик уже работает»

**Митигация:**
- Статус договора `INTENT` должен быть достаточен для создания Commitment
- Ретро-операция: возможность создать договор и Commitment задним числом без блокировки
- Юрист получает задачу параллельно, не блокируя оплату
- **Не делать договор hard-блокером на Фазе 0** — сделать мягким предупреждением

---

## 🔴 Риск #2: Гарантийное удержание — ручной ввод % (ВЕРОЯТНОСТЬ: ВЫСОКАЯ)

**Что происходит:**
`warranty_percent` задаётся при подписании договора. Если договор создаётся ретро-операцией или наспех — процент не вводится / вводится неправильно. Ledger считается без удержания. Субподрядчик получает больше, чем должен. Финансовая ошибка обнаруживается через 2-3 месяца.

**Признак провала:**
> «Лицевой счёт показывает неправильный баланс — мы переплатили субподрядчику»

**Митигация:**
- Блокировать подписание договора (INTENT → SIGNED) без заполнения `warranty_percent`
- Default = 5% (типичный для РК) с явным подтверждением
- В Ledger показывать отдельной строкой: «Гарантийное удержание: X ₸ (Y%)»

---

## 🟠 Риск #3: «Общий реестр Директора» — бутылочное горлышко (ВЕРОЯТНОСТЬ: ВЫСОКАЯ)

**Что происходит:**
Все субподрядные заявки летят напрямую к Директору. На большом проекте — 10-20 заявок в неделю. Директор перегружен, задерживает утверждение. Казначей не может платить. Субподрядчики злятся. Работы встают.

**Признак провала:**
> «Директор не успевает утверждать — заявки копятся по 2 недели»

**Митигация:**
- Лимитная матрица: суммы до X ₸ PM утверждает самостоятельно (ADR-18)
- Пакетное утверждение: «Утвердить все» с одним кликом
- SLA-уведомления: заявка ждёт > N дней → эскалация (ADR-27)
- **Это архитектурное решение нужно закрепить в Фазе 1, не потом**

---

## 🟠 Риск #4: OCR документов — неточность распознавания (ВЕРОЯТНОСТЬ: СРЕДНЯЯ)

**Что происходит:**
АВР от субподрядчика распознаётся с ошибками (сумма, БИН, позиции). Бухгалтер или ПТО не замечают — нажимают COMMITTED. Данные в Ledger неправильные. 1С видит расхождение (DISCREPANCY) через месяц.

**Признак провала:**
> «Система посадила неправильную сумму — пришлось исправлять вручную, но не знаем как»

**Митигация:**
- Поле уверенности OCR: если confidence < 85% → обязательная ручная проверка (подсветка)
- Корректировочные документы: возможность создать `AVR_ADJUSTMENT` без сброса статуса
- Разделение: `data_raw` (скан) vs `data_extracted` (поля) — редактируется только extracted
- **Нельзя запускать COMMITTED без проверки человеком — это не автоматический шаг**

---

## 🟡 Риск #5: Схема растёт вместе с пониманием (ВЕРОЯТНОСТЬ: СРЕДНЯЯ)

**Что происходит:**
В Фазе 1 создаётся схема для Subcontract. В Фазе 2 выясняется, что PO требует другое поле. В Фазе 3 SERVICE требует третье. Каждый раз — миграция, потенциальный Schema Lock конфликт. К Фазе 4 схема `commitment_items` содержит 40 полей, половина NULL для каждого типа.

**Признак провала:**
> «Схема стала God Table, нарушили ADR-38»

**Митигация:**
- С Фазы 0 ввести `commitment_type_metadata` (JSONB) для полей, специфичных для типа
- Якорь (`commitments`) = общие поля, type-specific таблицы = отдельные
- Schema Lock правило из AGENTS.md соблюдать строго
- Ревью схемы перед каждой новой фазой

---

## 🟡 Риск #6: L2-FIELD данные не вводятся (ВЕРОЯТНОСТЬ: СРЕДНЯЯ)

**Что происходит:**
Прораб должен ежедневно/еженедельно вносить прогресс в систему. На практике — вносит раз в месяц, задним числом, приблизительно. L2-D (биллинг) строится на этих данных → КС-2 содержит приблизительные данные → заказчик отклоняет → срыв выставления счёта.

**Признак провала:**
> «КС-2 не принимают, потому что цифры в системе не совпадают с фактом на объекте»

**Митигация:**
- Минимальный интерфейс для прораба: мобильная веб-страница, только % выполнения по строке ВОР
- Нотификации: «Заполни отчёт за неделю» каждую пятницу
- Ретро-ввод: прораб вводит данные за прошлые периоды — система помечает как `is_retrospective`
- **Не делать сложный интерфейс для прораба** — он не будет его использовать
