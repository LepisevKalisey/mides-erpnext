# Human Acceptance Test (HAT) — Слайс [B] (current_stage VIEW)

Этот документ описывает сценарии ручного тестирования для верификации вычисления статуса заявки через базу данных (`current_stage_view`) и его отображения в интерфейсе.

---

## Предусловия
1. База данных обновлена до миграции `0011_current_stage_view` (выполнено).
2. Запущен dev-сервер: `npm run dev` на порту `3000`.

---

## Сценарии тестирования

### Сценарий 1: Отображение статусов в реестре «Мои заявки»

1. Авторизуйтесь под ролью **Project Manager (PM)**.
2. Перейдите в раздел **«Мои заявки»** (`/dashboard/requests`).
3. Визуально проверьте колонку **«Статус»**:
   - Новые заявки без привязанных задач должны иметь серый бейдж **«Черновик»** (DRAFT).
   - Заявки, ожидающие согласования (есть `commitment_approval_tasks` в статусе `PENDING`), должны иметь желтый бейдж **«На рассмотрении»** (PENDING_APPROVAL).
   - Отклоненные директором заявки (`REJECTED`) — красный бейдж **«Отклонена»** (REJECTED).
   - Утвержденные директором заявки (`APPROVED`) — зеленый бейдж **«Утверждена»** (APPROVED).
   - Заявки, по которым прораб выполнил физприемку (`ACCEPTED`) — голубой бейдж **«Принята физически»** (ACCEPTED).
   - Заявки, по которым загружен АВР (`AVR_UPLOADED`) — сиреневый бейдж **«АВР загружен»** (AVR_UPLOADED).
   - Заявки с утвержденным АВР (`AVR_APPROVED`) — фиолетовый бейдж **«АВР утверждён»** (AVR_APPROVED).
   - Частично оплаченные казначеем (`PARTIAL` в `payment_tasks`) — синий бейдж **«Частично оплачена»** (PARTIALLY_PAID).
   - Полностью оплаченные (`PAID` в `payment_tasks`) — бирюзовый бейдж **«Оплачена»** (PAID).

*Ожидаемый результат:* Все бейджи отображают корректный перевод на русский язык, используют правильные семантические цвета из дизайн-системы и не содержат эмодзи.

---

### Сценарий 2: Фильтрация по вкладкам реестра

1. Перейдите во вкладку **«На рассмотрении»**:
   - Должны отображаться только заявки со статусом **«На рассмотрении»**.
   - Счётчик вкладки должен точно соответствовать количеству записей в списке.
2. Перейдите во вкладку **«Утверждённые»**:
   - В списке должны отображаться заявки со статусами: `APPROVED`, `ACCEPTED`, `AVR_UPLOADED`, `AVR_APPROVED`.
3. Перейдите во вкладку **«Оплаченные»**:
   - В списке должны отображаться только заявки со статусами: `PARTIALLY_PAID` и `PAID`.
4. Перейдите во вкладку **«Отклонённые»**:
   - В списке должны отображаться только заявки со статусом `REJECTED`.

*Ожидаемый результат:* Фильтрация работает без ошибок, счетчики вкладок реактивно обновляются и соответствуют фактическому наполнению.

---

### Сценарий 3: Защита от удаления согласованных заявок

1. Найдите заявку в статусе **«Черновик»** или **«На рассмотрении»**.
2. Убедитесь, что справа от даты создания присутствует иконка мусорной корзины (🗑) для удаления.
3. Найдите заявку в статусе **«Утверждена»** или **«Оплачена»**.
4. Убедитесь, что для этих заявок кнопка удаления отсутствует.

*Ожидаемый результат:* Удаление доступно только для заявок, которые еще не были утверждены или оплачены (DRAFT, PENDING_APPROVAL).

---

### Сценарий 4: Прямая проверка SQL VIEW (для инженеров)

1. Подключитесь к базе данных (через Supabase SQL Editor или консоль).
2. Выполните запрос:
   ```sql
   SELECT purchase_item_id, stage FROM current_stage_view LIMIT 10;
   ```
3. Убедитесь, что запрос возвращает плоскую таблицу соответствий ID заявок и их стадий, а значения в колонке `stage` соответствуют одному из перечислений: `DRAFT`, `PENDING_APPROVAL`, `APPROVED`, `REJECTED`, `ACCEPTED`, `AVR_UPLOADED`, `AVR_APPROVED`, `PARTIALLY_PAID`, `PAID`.

*Ожидаемый результат:* Запрос выполняется без ошибок за минимальное время (менее 50мс).
