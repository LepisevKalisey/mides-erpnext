# ADR-35: АВР-модуль — архитектура данных

**Статус:** Принято  
**Дата:** 2026-05-13  
**Заменяет:** ADR-23 (частично)  
**Дополняет:** ADR-15, ADR-12, ADR-21

---

## Контекст

Предыдущая модель (ADR-23) предполагала 2 страницы документа и последовательный роутинг.
После изучения реальных казахстанских форм (Р-1, КС-2, КС-3) и уточнения бизнес-процесса
архитектура была полностью пересмотрена.

---

## Документы пакета АВР

Каждый из трёх документов необязателен. Они накапливаются в разные таблицы.

| Форма | Листов | Суть | Таблица |
|-------|--------|------|---------|
| **Р-1** (Форма Р-1 Минфин РК) | 1 | Упрощённый АВР для услуг. Лампсумм: наименование, ед., кол-во, цена, итого, НДС | `r1_rows` |
| **КС-2** | 1..N | Детальный акт приёмки. Иерархия разделов, марки, колонки: по договору / до отч.месяца / в отч.месяце / накопительно | `ks2_rows` |
| **КС-3** | 1 | Справка о стоимости. 1 строка «Всего по стройке», 10 колонок с начала года / месяц / СМР / НДС / итого | `ks3_rows` |

Форматы загрузки: **PDF** → OCR, **Excel/CSV** → column-mapping UI без OCR.

---

## Принятые решения

### 1. Форма загрузки — два режима

**АВР:** только поле загрузки файлов. Контрагент, договор, проект, объект, период —
извлекаются из документов. Пользователь не выбирает вручную.

**Аванс:** ручной ввод — проект, объект, контрагент, сумма. Документов нет.
Проходит через `/approvals` директора идентично `PAYMENT_APPLICATION`.

Доступ: `users.works_with_contractors = true`.

### 2. Роутинг проверки — параллельный

ПТО и Бухгалтерия уведомляются одновременно. Каждый видит статус другого.
Каждый подтверждает независимо. `FULLY_APPROVED` — когда оба одобрили.

| Отдел | Видит | Задача |
|-------|-------|--------|
| **ПТО** | КС-2 | Проверка объёмов работ |
| **Бухгалтерия** | КС-3 + Р-1 | Сверка с историей платежей |

### 3. Матчинг КС-2 → спецификация договора — Late Binding

КС-2 хранится как есть. ПТО связывает строки КС-2 с позициями `contract_items`
через таблицу `ks2_row_links`. Накопительный итог — через VIEW.

Альтернативы отклонены:
- Прямое обновление `contract_items` — нет аудита
- Авто-матчинг (fuzzy) — реализуется позже, после накопления данных

### 4. Excel column-mapping

При загрузке xlsx/csv показывается UI сопоставления колонок.
Маппинги сохраняются per-contractor — при повторной загрузке применяются автоматически.

### 5. Таксономия itemType

```
MATERIALS  — расходуемые материалы (OPEX)
EQUIPMENT  — оборудование в ОС (CAPEX)
RENTAL     — аренда техники по маш.сменам (OPEX) [новый]
SERVICES   — подрядные услуги (OPEX)

Удалены: OFFICE (= projectType OVERHEAD), OTHER (антипаттерн)
```

---

## Жизненный цикл

```
DRAFT
  → OCR_PENDING / PROCESSING (Excel)
  → UPLOADER_REVIEW   ← загрузчик проверяет 3 вкладки, правит OCR
  → IN_REVIEW         ← ПТО + Бухгалтерия параллельно
  → FULLY_APPROVED    ← оба одобрили
  → ESF_PENDING → ESF_RECEIVED → EXPENSE_POSTED  (из ADR-15)
  → REJECTED          ← любой отдел может отклонить с причиной
```

---

## Последствия

**Положительные:**
- Р-1, КС-2, КС-3 накапливаются независимо — гибкость для неполных пакетов
- ks2_row_links сохраняет аудит трейл маппинга
- Параллельный роутинг ускоряет процесс
- Excel без OCR — надёжнее для структурированных файлов

**Отрицательные / риски:**
- OCR КС-2 (многостраничный, иерархический) — сложная задача парсинга
- Column-mapping UI требует отдельного цикла разработки
- Маппинг КС-2 → contract_items ложится на ПТО вручную
