# ADR-29: Жизненный цикл проекта

**Статус:** Принято  
**Дата:** 2026-05-11  
**Контекст:** Проектирование экрана создания проекта и логики его появления в системе

---

## Контекст

Директор ведёт переговоры с потенциальным заказчиком. Заказчик присылает материалы для оценки
и формирования КП. В этот момент проект должен появиться в системе как сущность — до подписания
договора, до назначения команды, до формирования ВОР.

Текущая схема (`project_status`) начиналась с `BIDDING` как минимального состояния.
Это не покрывало преддоговорные стадии и постпроектные обязательства (гарантийный период).

---

## Решения

### R-01: Полный жизненный цикл проекта

```
LEAD        → ESTIMATION → BIDDING → ACTIVE → CLOSEOUT → WARRANTY → CLOSED
                                                       ↘ SUSPENDED (из любого состояния)
```

| Статус | Кто переводит | Триггер |
|--------|---------------|---------|
| `LEAD` | DIRECTOR | Получены материалы от заказчика для оценки |
| `ESTIMATION` | DIRECTOR | Назначен хотя бы один ответственный (PM или PTO_ENGINEER) |
| `BIDDING` | DIRECTOR / PM | КП/тендерное предложение подано заказчику |
| `ACTIVE` | DIRECTOR | Договор с заказчиком подписан |
| `CLOSEOUT` | PM | Работы физически завершены, идут финансовые закрытия |
| `WARRANTY` | PM / ACCOUNTANT | Подписан КС-3, объект передан заказчику, начался гарантийный период |
| `CLOSED` | DIRECTOR | Гарантийные обязательства истекли, всё закрыто |
| `SUSPENDED` | DIRECTOR | Заморожен (решение заказчика, финансирование, форс-мажор) |

**Обоснование `WARRANTY` отдельно от `CLOSED`:**
После сдачи объекта заказчику МиДЭС несёт гарантийные обязательства (обычно 2–5 лет).
В этот период может быть удержан гарантийный депозит, возникают гарантийные случаи.
`CLOSED` означает полное прекращение всех обязательств.

### R-02: Кто создаёт проект

**Только DIRECTOR** (или ADMIN).

Создание проекта даже в статусе `LEAD` — управленческое решение:
«мы рассматриваем этот заказ». Это не техническая операция.

Экран: Дашборд директора → раздел «Проекты» → кнопка «Новый проект».

### R-03: Минимальные обязательные данные при создании (LEAD)

| Поле | Тип | Обязательно | Примечание |
|------|-----|-------------|------------|
| `name` | text | ✅ | Рабочее название: «ЖК Мирас — Паркинг» |
| `client_name` | text | ✅ | Может быть условным, уточняется позже |
| `contact_person` | text | ✅ | ФИО контактного лица заказчика |
| `contact_info` | text | ✅ | Телефон / email / Telegram |
| `note` | text | ❌ | Контекст, пожелания, сроки оценки |

Всё остальное (`contract_number`, `contract_amount`, `start_date` и т.д.) — заполняется позже.

**Обоснование обязательности контакта:**  
Контактное лицо нужно немедленно — ПТО или PM должны иметь возможность задать
уточняющие вопросы при оценке. Без контакта проект «подвисает» в неопределённости.

### R-04: Видимость проекта для других пользователей

| Статус | Кто видит |
|--------|-----------|
| `LEAD` | Только DIRECTOR и ADMIN |
| `ESTIMATION` и далее | DIRECTOR + ADMIN + все назначенные через `project_assignments` |
| `ACTIVE` и далее | DIRECTOR + ADMIN + назначенные + ECONOMIST (видит все активные проекты) |

**Триггер расширения видимости:** первое назначение в `project_assignments`
автоматически переводит проект из `LEAD` → `ESTIMATION`.

Это означает: директор сначала создаёт проект (LEAD), затем назначает ответственного
(например, PM или PTO_ENGINEER для оценки) — и только тогда проект становится
виден команде.

## Последствия

### Изменения схемы БД

**Enum `project_status`** — добавить:
- `LEAD` (до `BIDDING`)
- `ESTIMATION` (до `BIDDING`)
- `WARRANTY` (после `CLOSEOUT`)

**Таблица `projects`** — добавить поля:
- `contact_person text NOT NULL` — ФИО контакта
- `contact_info text NOT NULL` — телефон/email

### Изменения бизнес-логики

- При создании проекта: статус = `LEAD` (не `ACTIVE` как сейчас)
- При первом `project_assignments.insert` → триггер/логика в server action: 
  если `project.status === 'LEAD'` → обновить до `'ESTIMATION'`
- Фильтр видимости: применять на уровне server actions (не RLS, т.к. отключён)

### Изменения UI

- Форма создания проекта: 4 обязательных поля + note
- Список проектов директора: показывать все статусы включая LEAD
- Список проектов для остальных: фильтровать — только где есть их assignment
