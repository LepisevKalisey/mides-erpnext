# ADR-02: Навигация по отделам

**Статус:** Accepted  
**Дата:** 2026-05-04  
**Автор:** Архитектор системы

---

## Контекст

Навигация должна отражать организационную структуру компании. Экраны группируются по отделам. Сотрудник видит только те группы и экраны, к которым ему назначен доступ.

## Решение

### Организационная структура (зафиксированная)

```
1. Финансы
   ├── Бухгалтерия
   ├── Кадры
   └── Казначейство        ← первый реализуемый экран
2. Обеспечение
   ├── Поставки оборудования
   └── Снабжение
3. Администрация
   ├── HR
   └── Юр.отдел
4. Производство
   ├── ПТО
   └── Линия
5. Коммерческий отдел
```

### Структура навигации

Левое меню строится **динамически** из таблицы `screens`, отфильтрованной по ролям пользователя:

```ts
// Псевдокод построения навигации
const userScreens = await db.select(screens)
  .join(screenAccess, eq(screens.key, screenAccess.screenKey))
  .where(inArray(screenAccess.role, userRoles))
  .orderBy(screens.department, screens.subDepartment)

// Группировка по department → subDepartment → screens[]
const navTree = groupBy(userScreens, 'department', 'subDepartment')
```

### Принципы навигации

1. **Рабочий день** — первая вкладка на экране каждого отдела. Универсальный шаблон, наполняемый данными конкретного процесса. В будущем сюда подключается task engine.
2. Группы отделов коллапсируются если недоступны пользователю
3. Активный пункт подсвечивается по URL (`pathname`)
4. Алгоритм расчёта для сайдбара выполняется на сервере (layout.tsx), передаётся клиентскому компоненту

### Ключ экрана

Формат: `{отдел}.{под-отдел}.{функция}`

Примеры:
- `finance.treasury.workday`
- `finance.treasury.summary`
- `finance.treasury.registries`
- `legal.contracts.list`
- `admin.refs` (справочники)

## Последствия

- Сайдбар становится data-driven (данные из БД)
- При добавлении нового экрана: 1 строка в `screens` + запись в `screen_access`
- Навигация не изменяется при реорганизации отделов — меняется только `department` в таблице `screens`
