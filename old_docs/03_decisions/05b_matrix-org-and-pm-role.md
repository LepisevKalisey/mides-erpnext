# ADR-05: Матричная оргструктура МиДЭС

**Статус:** Принято  
**Дата:** 2026-05-10  
**Версия:** 3.0 — финальная модель

---

## Контекст

МиДЭС ведёт несколько строительных проектов одновременно. Каждый проект требует
участия сотрудников из разных отделов (снабжение, ПТО, производство). При этом:

- На разных объектах технический руководитель разный
- Нет единого ответственного за коммерческую сторону проекта
- Сотрудник может работать на нескольких проектах одновременно
- Нагрузка на сотрудников неравномерна и не видна руководителям

Это классическая потребность в **матричной организации**.

---

## Принятая модель: Сбалансированная матрица

### Два уровня: компания и проект

---

## Уровень 1 — Постоянная иерархия компании

Определяет кто кому подчиняется, карьерное развитие, кадровые решения.

```
ГЕНЕРАЛЬНЫЙ ДИРЕКТОР
  ├── DEPUTY_DIRECTOR [CIVIL]     (Зам по общестрою)
  │     └── SITE_SUPERINTENDENT × N  (Прорабы — общестрой)
  │
  ├── DEPUTY_DIRECTOR [MEP]       (Зам по инженерным сетям)
  │     └── SITE_SUPERINTENDENT × N  (Прорабы — инж.сети)
  │
  ├── DEPUTY_DIRECTOR [GENERAL]   (Зам по общим вопросам)
  │     ├── HR
  │     ├── IT
  │     └── OFFICE_MANAGER
  │
  ├── HEAD_OF_PROCUREMENT
  │     ├── WAREHOUSE × N         (Кладовщики)
  │     └── BUYER × N             (Снабженцы)
  │
  ├── HEAD_OF_PTO
  │     └── PTO_ENGINEER × N
  │
  ├── ECONOMIST                   (подчиняется DIRECTOR напрямую)
  │
  ├── HEAD_OF_ACCOUNTANTS
  │     └── ACCOUNTANT × N
  │
  └── TREASURER
```

**Правило:** постоянная иерархия не меняется при назначении на проект.
Прораб остаётся подчинённым своего Зама даже будучи Начальником участка на объекте.

---

## Уровень 2 — Иерархия проектной команды

### Трёхуровневая структура: Проект → Объект → Направление

```
ПРОЕКТ "Строительство школы Мирас"      ← весь договор
  ├── ОБЪЕКТ "Блок А"                    ← физическое подразделение
  │     ├── Направление: Общестрой (CIVIL)
  │     └── Направление: Инж.сети (MEP)
  └── ОБЪЕКТ "Блок Б"
        ├── Направление: Общестрой (CIVIL)
        └── Направление: Инж.сети (MEP)
```

**Важно:** «Проект» = договор. «Объект» = физическая часть стройки.
Смешивать эти понятия нельзя.

### Структура команды на проекте

```
ПРОЕКТ X
│
├── ГИ проекта                          (технический домен)
│     │
│     ├── Инженер ПТО
│     │     └── ведёт ВОР, маппинг, исполнительную документацию
│     │
│     └── Начальник участка [CIVIL]     (SITE_MANAGER)
│     └── Начальник участка [MEP]       (SITE_MANAGER)
│           │
│           ├── Кладовщик               (выдача материалов субчикам)
│           ├── Прораб
│           │     └── Мастер
│           │
│           └── ← СУБПОДРЯДЧИКИ
│                 (работают физически здесь, принимает Начучастка)
│
└── ПМ проекта                          (коммерческий домен)
      │
      ├── Снабженец
      ├── Бухгалтер
      └── Экономист
```

## Субподрядчики: единая точка контакта и рычаги давления

### Правило для субчика: «все вопросы — Начальнику участка»

Субчик не разделяет технические и коммерческие вопросы.
Его единственный контакт — **Начальник участка** на объекте.
ПМ управляет деньгами, но деньги появляются только после приёмки Начучастком.

### Рычаги давления Начальника участка

| Рычаг | Механизм в системе |
|---|---|
| **Физическая приёмка** | `physical_acceptance` обязательна перед созданием АВР. Без подписи Начучастка АВР не существует |
| **Частичная приёмка** | Принимает 70м² из 100м² — оплата только за принятое |
| **Дефектная ведомость** | Фиксирует замечания в системе, блокирует приёмку до устранения |
| **Выдача материалов** | Кладовщик подчинён Начучастку — может придержать материалы до исправления работ |

### Маршрут АВР субподрядчика

```
Субчик выполнил работу
    ↓
SITE_MANAGER: физическая приёмка (фото, объёмы, замечания)
    ↓ отклонено → субчик получает дефектную ведомость, устраняет
    ↓ принято
ГИ: технический gate («объёмы верны, качество ОК»)
    ↓
ПМ: коммерческий gate («в рамках договора и бюджета»)
    ↓
DIRECTOR: финальное утверждение (или авто ниже лимита)
    ↓
ACCOUNTANT проекта: физическое получение документов, оплата
```

**ПМ не может одобрить АВР без принятой приёмки от SITE_MANAGER** — это структурный запрет системы.

## Проектные роли и маршрутизация

### Таблица проектных ролей

| project_role | Домен | Назначается из | Ключевая функция |
|---|---|---|---|
| `GI` | Технический | DEPUTY_DIRECTOR или опытный PTO_ENGINEER | Технический gate АВР |
| `PM` | Коммерческий | Любой, по решению DIRECTOR | Коммерческий gate АВР |
| `SITE_MANAGER` | Производство | SITE_SUPERINTENDENT | Физическая приёмка, управление субчиками |
| `PTO_ENGINEER` | Документация | PTO_ENGINEER | ВОР, маппинг, КС-2 |
| `BUYER` | Снабжение | BUYER | Закупки проекта |
| `ACCOUNTANT` | Финансы | ACCOUNTANT | Документы проекта, оплаты |
| `WAREHOUSE` | Склад | WAREHOUSE | Приёмка GRN, выдача материалов |

### Маршрутизация по направлению

На одном объекте — два Начальника участка:

```
АВР от субчика содержит: object_id + work_stream (CIVIL | MEP)

Система ищет:
  project_assignments WHERE object_id=X AND work_stream=CIVIL AND project_role=SITE_MANAGER
  → направляет нужному Начальнику участка
```

## Со-назначение и управление нагрузкой

На одну проектную роль можно назначить несколько человек (co-assignment).
**OR-логика:** первый кто действует — документ обработан.

### Кто добавляет со-назначенного

| Роль | Может назначить своих |
|---|---|
| `DIRECTOR` | Кого угодно на любую роль |
| `DEPUTY_DIRECTOR` | Своих SITE_SUPERINTENDENT |
| `HEAD_OF_PROCUREMENT` | Своих BUYER и WAREHOUSE |
| `HEAD_OF_PTO` | Своих PTO_ENGINEER |
| `HEAD_OF_ACCOUNTANTS` | Своих ACCOUNTANT |

### Видимость нагрузки

Руководитель видит таблицу подчинённых:

```
Имя          Проектов   Ожидает решения   Сигнал
Иванов          3            8              🔴 высокая
Петров          1            2              🟢 свободен
Смирнов         2            4              🟡 средняя
```

## Таблицы данных

```sql
-- Трёхуровневая иерархия
projects (
  id, name, contract_id, client_id, status, ...
)

project_objects (
  id, project_id, name, ...          -- Блок А, Блок Б
)

-- Назначения команды
project_assignments (
  id,
  project_id,
  object_id,         -- NULL = весь проект; NOT NULL = конкретный объект
  work_stream,       -- NULL | 'CIVIL' | 'MEP'
  user_id,
  project_role,      -- GI | PM | SITE_MANAGER | PTO_ENGINEER | ...
  domain,            -- 'TECHNICAL' | 'COMMERCIAL' (вычисляется из project_role)
  is_primary,
  assigned_by,
  assigned_at,
  removed_at,
  reason
)

-- Физическая приёмка (создаётся SITE_MANAGER перед АВР)
physical_acceptances (
  id,
  project_id,
  object_id,
  work_stream,
  subcontractor_id,
  accepted_by,         -- SITE_MANAGER
  accepted_qty,
  defects_noted,       -- JSON: список замечаний
  photos,              -- ссылки на фото
  status,              -- ACCEPTED | PARTIAL | REJECTED | PENDING_FIX
  accepted_at
)

-- АВР всегда ссылается на приёмку
avr_documents (
  id,
  physical_acceptance_id,  -- ОБЯЗАТЕЛЬНОЕ поле, FK → physical_acceptances
  ...
)
```

## Фазированное внедрение

**Фаза 1 (текущий релиз):**
- `projects` + `project_objects` (Проект → Объект)
- `project_assignments` с `object_id` и `work_stream`
- `physical_acceptances` как обязательный шаг перед АВР
- Маршрут: SITE_MANAGER → GI → PM → DIRECTOR
- Дашборд нагрузки для функциональных руководителей

**Фаза 2:**
- Дефектная ведомость с фото и статусом устранения
- Частичная приёмка с накоплением по периодам
- Субподрядчик видит статус своей приёмки и АВР

**Фаза 3 (task-engine):**
- Реальная нагрузка по задачам и часам
- Конфликты назначений

---

## Последствия

**Позитивные:**
- SITE_MANAGER сохраняет полный рычаг над субчиками через physical_acceptance
- Чёткое разделение: технический домен (GI) / коммерческий домен (PM)
- Субчик знает одно правило: «все вопросы — Начальнику участка»
- Маршрутизация АВР автоматична: по объекту и направлению

**Риски:**
- SITE_MANAGER может злоупотребить задержкой приёмки (нужна эскалация к GI)
- Сложность маршрутизации при нескольких SITE_MANAGER на объекте
