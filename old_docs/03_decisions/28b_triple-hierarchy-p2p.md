# ADR-28: Тройная иерархия в P2P (Проект → Объект → Направление)

**Статус:** Принято  
**Дата:** 2026-05-11  
**Связано с:** ADR-10 (project-entity), ADR-12 (purchase-item-lifecycle)

---

## Контекст

Ранее существовавшая двойная иерархия (Проект → Объект) не позволяла достаточно гранулярно разделять затраты и ответственность внутри одного физического объекта. В строительстве критически важно разделять общестроительные работы (CIVIL) и инженерные сети (MEP), так как они имеют:
- Разные бюджетные статьи.
- Разных ответственных (Supply Managers / Project Managers).
- Разных поставщиков.

## Решение

Внедряется обязательная тройная структура для всех позиций закупа (Purchase Items) и связанных сущностей:

1.  **Project (Проект):** Коммерческая единица (договор с заказчиком).
2.  **Object (Объект):** Физическое место производства работ (участок, пятно застройки).
3.  **Work Stream / Direction (Направление):** Функциональная область работ.
    - `CIVIL` — Общестроительные работы.
    - `MEP` — Инженерные сети (механика, электрика, сантехника).

### Изменения в схеме данных

В таблицу `midescloud.purchase_items` добавлено поле:
- `work_stream` (ENUM: `CIVIL`, `MEP`).

### Изменения в UI/UX

1.  **Форма создания заявки:** При выборе объекта пользователь обязан указать направление работ.
2.  **Реестр заявок (Registry):** В основной таблице добавлена колонка «Направление», позволяющая фильтровать закупки по специализации.
3.  **Роутинг:** Правила согласования теперь могут учитывать направление (например, разные снабженцы для CIVIL и MEP на одном объекте).

## Последствия

**Положительные:**
- Точный учёт затрат по направлениям работ.
- Возможность автоматического назначения ответственных снабженцев на основе специализации.
- Упрощение сверки с бюджетом, который часто разбит по разделам CIVIL/MEP.

**Отрицательные:**
- Усложнение формы ввода (лишний клик для инициатора).
- Необходимость классификации существующих данных (миграция).

## Альтернативы

- **M2M связь Объект-Направление:** Избыточно, так как направление — это свойство конкретной работы/закупки, а не объекта.
- **Использование Cost Codes:** Слишком сложно для линейного персонала на этапе подачи заявки. Направление — понятный верхнеуровневый фильтр.
