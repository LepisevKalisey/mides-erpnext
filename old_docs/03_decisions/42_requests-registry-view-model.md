# ADR-042: Модель представлений реестра заявок (Requests & Registry View Model)

**Статус:** Утверждён  
**Дата:** 2026-05-19  
**Связан с:** ADR-36 (unified-payment-registry), ADR-27 (sla-framework), ADR-15 (avr-lifecycle-esf), ADR-38 (task-per-role), ADR-31 (org-structure-and-roles), ADR-14 (overhead-project-type), ADR-18 (approval-matrix)

---

## Контекст

Единый реестр `purchase_items` (ADR-36) виден разным ролям в разном контексте.
Нужно определить: как организовать маршруты и экраны так, чтобы каждая роль видела
ровно то, что нужно для работы — без лишнего шума и без дублирования данных.

### Ключевые наблюдения

**Все роли без исключения являются создателями** — у каждого есть личные заявки.
«Операторские» роли дополнительно работают с заявками всей компании на отдельных экранах.

**Роль Директора при создании заявки:**
Директор создаёт обычную `PAYMENT_APPLICATION`. Никакого специального `commitment_type`
не нужно — его заявка получает статус `APPROVED` автоматически при создании, так как
он сам является утверждающим. Это использует стандартный статусный механизм,
а не плодит новые сущности.

---

## Решение

### Принцип: Разделение контекстов, не данных

```
ОДИН источник данных: purchase_items
ДВА типа экрана:
  1. Личный ("Мои заявки")   — WHERE created_by = me
  2. Операционный (реестр)   — WHERE stage = X AND role_filter
```

### Маршруты и экраны

```
/requests                  — "Мои заявки"         (все роли)
/requests/approvals        — Очередь утверждений  (Директор / Зам. директора)
/payments                  — Казначейский стол     (Казначей)
/accounting                — Рабочее место         (Бухгалтерия)
/sourcing                  — Рабочее место         (Отдел снабжения)
```

> `/payments`, `/accounting`, `/sourcing` — самостоятельные операционные рабочие
> места, а не подразделы `/requests`.

---

### Детали каждого экрана

#### /requests — "Мои заявки" (все роли)

```
Данные:    purchase_items WHERE created_by = current_user
Действия:  + Создать заявку (role-aware sheet)
Колонки:   Описание, Проект, Сумма, Дата, Статус (badge)
```

**Типы заявок, доступные при создании (role-aware):**

| Роль | Доступные типы при нажатии «+» |
|------|-------------------------------|
| ПМ | АВР субподряда, Аванс субподрядчику, Запрос материалов |
| Прораб / Нач. участка | Запрос материалов |
| Любой офисный сотрудник | Запрос оборудования, Аванс на услуги |
| **Директор** | Платёжное поручение (авто-APPROVED) |
| Казначей | Запрос оборудования, Аванс на услуги |
| Снабженец | Запрос оборудования |
| Бухгалтер | Запрос оборудования, Аванс на услуги |

**Инвариант — заявка Директора:**
- Создаётся как `PAYMENT_APPLICATION`, `flow_type = 'PAYMENT_APPLICATION'`
- При сохранении `commitmentApprovalTask.status` сразу выставляется `APPROVED`
- Директор не проходит через чужую очередь утверждения — он сам утверждающий
- **Никакого специального `commitment_type = DIRECT_ORDER` не создаётся**

**Инвариант — дефолтный проект:**
- Сотрудник без привязки к строительному проекту → `project_id = OFFICE_PROJECT`
- Офисные расходы прозрачны: видны в том же реестре с `project.type = OVERHEAD` (ADR-14)

---

#### /requests/approvals — Очередь утверждений (Директор / Зам.)

```
Данные:    purchase_items
           JOIN commitment_approval_tasks t ON t.purchase_item_id = pi.id
           WHERE t.approverId = current_user AND t.status = 'PENDING'
Действия:  Утвердить, Отклонить (с обязательным комментарием)
Колонки:   Автор, Проект/Объект, Подрядчик, Сумма, Дата, Тип заявки, SLA-таймер
```

Директор видит только чужие заявки, ожидающие его подписи.
Его собственные заявки — в `/requests`, не здесь.

SLA-таймер: `APPROVAL_WAIT` = 3 раб. дня (ADR-27).

---

#### /payments — Казначейский стол (Казначей)

```
Данные:    purchase_items WHERE approvalStatus = 'APPROVED'
           (все одобренные заявки, готовые к оплате)
Действия:  Создать платёж, Привязать к банковскому счёту, Назначить дату
Колонки:   Проект, Подрядчик, Договор, Сумма, Утверждено кем/когда, Приоритет
```

Казначей создаёт свои личные заявки через `/requests`.
На `/payments` он работает только с чужими одобренными заявками.

---

#### /accounting — Рабочее место бухгалтерии (Бухгалтерия)

Источник: ADR-15 (avr-lifecycle-esf), ADR-38 (`accounting_tasks`).

```
Данные:    accounting_tasks JOIN purchase_items
           (задачи: проверка АВР, мониторинг ESF, проводка в 1С)
Действия:  Согласовать АВР, Отметить ESF получен, Подтвердить проводку
Колонки:   Тип, Проект, Подрядчик, Сумма, Шаг, Дедлайн ESF, Статус
```

**Специфика бухгалтерии (ADR-15):**
- Участвует в согласовании АВР: подписание на шаге `AVR_REVIEW` (SLA 2 раб. дня)
- Мониторинг ESF от субподрядчика: 30 кал. дней на получение
- Проводка расхода после получения ESF: `ESF_RECEIVED → EXPENSE_POSTED`
- Интеграция с 1С: бухгалтерия не управляет данными в 1С напрямую — MidesCloud
  является источником истины, 1С получает события (ADR-16)

Бухгалтер создаёт свои личные заявки через `/requests`.
На `/accounting` — только операционная работа с задачами всей компании.

---

#### /sourcing — Рабочее место снабжения (Отдел, не один человек)

Источник: ADR-27 (sla-framework §Управление загрузкой снабжения), ADR-28 (CIVIL/MEP).

Снабжение — **отдел**. Есть две роли:
- `PROCUREMENT` — рядовой снабженец
- `PROCUREMENT_LEAD` — начальник отдела (`PROCUREMENT` + `is_team_lead = true`)

**Три потока заявок для снабженца:**

| Поток | Фильтр | Кто видит |
|-------|--------|-----------|
| **Пул** (ничейные) | `flow_type = 'PROCUREMENT' AND stage = 'POOL_AVAILABLE'` | Все снабженцы |
| **Мои объекты** | `project_assignments WHERE user_id = me AND project_role = 'BUYER'` | Назначенный снабженец |
| **Назначены на меня** | `assigned_to = me` | Назначенный снабженец |

**Действия снабженца:**
- Взять заявку из пула (self-assign) → `POOL_AVAILABLE → IN_SOURCING`
- Работать с назначенными заявками: выбор поставщика, запрос КП, создание PO

**Дополнительные возможности PROCUREMENT_LEAD:**

| Действие | PROCUREMENT | PROCUREMENT_LEAD |
|---|:---:|:---:|
| Взять заявку из пула | ✅ | ✅ |
| Назначить заявку другому снабженцу | — | ✅ |
| Видеть все заявки команды | — | ✅ |
| Видеть загрузку каждого сотрудника | — | ✅ |
| Перераспределить задачи | — | ✅ |

**SLA (ADR-27):**
- `POOL_ASSIGNMENT` = 16 раб. часов (заявка без назначения)
- `SOURCING` = 3 раб. дня (от назначения до выбора поставщика)

Разделение CIVIL/MEP (ADR-28): снабженец видит заявки по своей специализации
(`project_assignments.work_stream`). PROCUREMENT_LEAD видит всё.

---

### Архитектура компонента

Все пять экранов используют **один переиспользуемый компонент** `<PurchaseItemsGrid />`:

```typescript
// ОДИН AccordionGrid компонент — разные конфигурации
<PurchaseItemsGrid
  query={queryType}     // 'mine' | 'approvals' | 'payments' | 'accounting' | 'sourcing'
  showActions={actions} // какие кнопки показывать
  columns={colSet}      // какие колонки показывать
/>
```

## Инварианты

1. **Источник данных один**: все экраны читают из `purchase_items`
2. **"Мои заявки" доступны всем**: каждый сотрудник может создать заявку
3. **Дефолтный проект**: сотрудник без проекта → OFFICE_PROJECT (ADR-14)
4. **Заявка Директора**: обычная `PAYMENT_APPLICATION`, автоматически `APPROVED` при создании — без нового `commitment_type`
5. **Снабжение = отдел**: пул заявок виден всем снабженцам, назначение — через self-assign или PROCUREMENT_LEAD
6. **Бухгалтерия**: работает с `accounting_tasks`, не с `purchase_items` напрямую
7. **Доступ к операторским экранам**: middleware проверяет роль; нет роли — редирект на `/requests`

## Схема навигации (sidebar, role-aware)

```
ПМ / Прораб / Офис:
  ├── 📋 Мои заявки         /requests

Директор / Зам.:
  ├── 📋 Мои заявки         /requests
  └── ✅ Утверждения         /requests/approvals  ← badge с количеством PENDING

Казначей:
  ├── 📋 Мои заявки         /requests
  └── 💳 Казначейский стол  /payments

Бухгалтер:
  ├── 📋 Мои заявки         /requests
  └── 📂 Бухгалтерия        /accounting

Снабженец (PROCUREMENT):
  ├── 📋 Мои заявки         /requests
  └── 📦 Снабжение          /sourcing  (пул + мои объекты + назначенные мне)

Начальник снабжения (PROCUREMENT_LEAD):
  ├── 📋 Мои заявки         /requests
  └── 📦 Снабжение          /sourcing  (+ загрузка команды + перераспределение)
```

---

## Альтернативы

| Альтернатива | Почему отклонена |
|---|---|
| `commitment_type = DIRECT_ORDER` для Директора | Плодит сущности без необходимости; стандартный `APPROVED` статус решает задачу |
| Один `/requests` с табами | Директор/Казначей видят таб "Мои заявки" с нулём своих заявок; операционный реестр — не подтаб личных заявок |
| Отдельные данные для каждого экрана | Нарушает ADR-36; дублирование логики |
| `/commitments` как маршрут | Слишком узко (только субподряд); `/requests` покрывает все типы |
