# ADR-33: Принцип каскадной видимости

**Статус:** Принято  
**Дата:** 2026-05-13  
**Связано с:** ADR-13 (routing), ADR-29 (project-lifecycle), ADR-30 (object-as-child), ADR-31 (org-roles)

---

## Контекст

В системе существует иерархия сущностей:

```
Компания
  └── Проект
        └── Объект
              └── Позиция / Заявка / Задача
```

При назначении сотрудника на уровне иерархии возникает вопрос: на какие нижестоящие сущности он получает права и видимость?

---

## Решение

**Принцип каскадной видимости:**

> Если сотруднику не задано ограничение на нижестоящий уровень иерархии —  
> он автоматически видит **все** сущности этого уровня.

### Применение

| Назначение | Видит |
|-----------|-------|
| Назначен на **компанию** (роль без проекта) | Все проекты, все объекты, все позиции |
| Назначен на **проект** (object_id IS NULL) | Все объекты проекта, все позиции всех объектов |
| Назначен на **объект** конкретного проекта | Только этот объект и его позиции |
| Назначен на **объект** (без project_id) | Не используется — object всегда принадлежит проекту |

### Примеры

```
Снабженец Иван назначен на Проект «ЖК Мирас», object_id = NULL
→ Иван видит позиции всех объектов: Корпус А, Корпус Б, Паркинг, Наружные сети

Снабженец Арман назначен на Проект «ЖК Мирас», object_id = «Корпус А»
→ Арман видит только позиции Корпуса А

Начальник снабжения назначен на уровне компании (ADMIN/HEAD_OF_PROCUREMENT)
→ Видит все позиции всех проектов
```

## Реализация в запросах

```sql
-- Для роутинга: найти BUYER для позиции
SELECT pa.user_id
FROM project_assignments pa
WHERE pa.project_id = :project_id
  AND pa.project_role = 'BUYER'
  AND (pa.object_id = :object_id OR pa.object_id IS NULL)
ORDER BY pa.object_id NULLS LAST  -- точное совпадение имеет приоритет
```

```sql
-- Для фильтрации: какие позиции видит снабженец
SELECT pi.*
FROM purchase_items pi
WHERE pi.project_id = ANY(
  -- проекты, где снабженец назначен
  SELECT project_id FROM project_assignments
  WHERE user_id = :user_id AND project_role = 'BUYER'
)
AND (
  -- без привязки к объекту → все объекты проекта
  EXISTS(
    SELECT 1 FROM project_assignments
    WHERE user_id = :user_id AND project_id = pi.project_id AND object_id IS NULL
  )
  OR
  -- с привязкой → только свой объект
  pi.object_id IN (
    SELECT object_id FROM project_assignments
    WHERE user_id = :user_id AND project_id = pi.project_id AND object_id IS NOT NULL
  )
)
```

## Распространение принципа

Принцип применяется **ко всем отделам и ролям**:

- **ПМ без object_id** → видит все объекты проекта
- **Прораб (SITE_SUPERINTENDENT) с object_id** → только свой объект
- **ПТО-инженер без object_id** → все объекты проекта
- **Бухгалтер без object_id** → все финансовые данные проекта
- **Начальник отдела (HEAD_*)** → видит распределение по проектам для своего отдела

---

## Управление командой

Начальники отделов (HEAD_OF_PROCUREMENT, HEAD_OF_ENGINEERING и т.д.) могут:
1. Видеть распределение своих сотрудников по проектам
2. Переназначать сотрудников внутри своего отдела
3. Усиливать команду (добавлять второго снабженца/инженера)
4. При изменении — уведомление всем участникам проекта

---

## Альтернативы

- **Явные permissions на каждый объект**: слишком детально, административная нагрузка высока
- **Flat список объектов без иерархии**: не масштабируется, нет ясной ответственности
