# ADR-03: Роль как позиция сотрудника

**Статус:** Accepted  
**Дата:** 2026-05-04  
**Автор:** Архитектор системы

---

## Контекст

Первоначально в системе была задумана роль EMPLOYEE — обобщённый сотрудник. В ходе архитектурного обсуждения выяснилось, что этой роли не существует в реальной организации: каждый сотрудник занимает конкретную позицию в конкретном отделе. Функции определяются позицией, а не фактом наличия аккаунта.

## Решение

### Роль = Позиция (должность) в отделе

Роль в системе — это не уровень доступа (не "read-only" / "editor"), а функциональная позиция сотрудника:
- `TREASURER` — Казначей (Финансы / Казначейство)
- `DIRECTOR` — Директор (общее руководство)
- `LAWYER` — Юрист (Администрация / Юр.отдел)
- `ADMIN` — Системный администратор

### Мультипозиционность

Один сотрудник **может иметь несколько позиций одновременно**. Это реализуется через junction-таблицу `user_roles` (уже существует). Типичный кейс: сотрудник временно покрывает задачи другого отдела — ему назначается дополнительная роль, а вместе с ней открываются экраны этого отдела.

```
Иванов Иван → [TREASURER, LAWYER]
  Видит: Казначейство + Договоры
```

### Что удалено

- `EMPLOYEE` — исключён из enum. Не существует как функциональная позиция.

### Расширение ролей

При добавлении новых отделов — добавляются новые значения в enum `role`:
- Будущие: `SUPPLY_MANAGER`, `ACCOUNTANT`, `PTO_ENGINEER` и т.д.
- Каждое новое значение требует: 1) миграции enum, 2) регистрации экранов в `screen_access`

## Последствия

- Удаление EMPLOYEE из `roleEnum` — требует миграции. Выполняется в первый спринт без ущерба (EMPLOYEE не назначалась реальным пользователям).
- Новые роли добавляются по мере введения новых отделов в систему.
- Администратор может назначить сотруднику несколько ролей через UI `/admin/users`.
