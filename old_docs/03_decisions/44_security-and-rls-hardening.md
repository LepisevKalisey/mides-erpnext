# ADR-44: Усиление безопасности и принудительное включение RLS

**Статус:** Утверждён  
**Дата:** 2026-05-20  
**Участники:** Разработчик, Архитектор, Безопасность  
**Связан с:** ADR-01 (Модель доступа), ADR-09 (Стек технологий), ADR-43 (Схема payment_tasks)

---

## Контекст

При внедрении «Казначейского стола» (Слайс 1.3) и создании таблицы `payment_tasks` возникли проблемы с правами доступа. По умолчанию в Supabase новые таблицы, создаваемые через миграции Drizzle, получают права SELECT/INSERT/UPDATE/DELETE для роли `anon` (неавторизованные пользователи), а Row Level Security (RLS) выключен.

Это создаёт серьёзную уязвимость: любой пользователь интернета, зная публичный `anon` ключ (который открыто передаётся на клиенте), может читать и перезаписывать данные в этих таблицах напрямую через PostgREST.

Также в системе была обнаружена ещё одна таблица с отключенным RLS — `legal_work_types`.

Необходимо реализовать принцип минимальных привилегий (Principle of Least Privilege) и защитить систему от несанкционированного доступа.

## Решение

Были приняты следующие меры по усилению безопасности и исправлению багов авторизации данных:

1. **Использование Row Level Security (RLS) вместо блокировки ролей**:
   Поскольку клиентские запросы PostgREST используют публичный API-ключ (ассоциированный с ролью `anon`), даже при наличии JWT-сессии авторизованного пользователя, СУБД выполняет запросы от имени роли `anon` (или `authenticated` в зависимости от конфигурации клиента Supabase). Блокировка роли `anon` через `REVOKE` приводила к тому, что запросы возвращали пустые данные (`0 ₸`) на клиенте.
   
   Вместо этого был выбран стандартный подход Supabase:
   - Права на `SELECT, INSERT, UPDATE, DELETE` выдаются как роли `authenticated`, так и роли `anon`.
   - Безопасность гарантируется принудительным включением RLS на всех 24 базовых таблицах.
   - Базовая политика RLS требует наличия валидной JWT-сессии: `USING (auth.uid() IS NOT NULL)`.
   
   Если пользователь не авторизован (`auth.uid()` равен `NULL`), RLS-политика возвращает пустой результат, обеспечивая абсолютную защиту данных.

2. **Принудительное включение RLS для всех таблиц**:
   Row Level Security включен для всех базовых таблиц системы (включая `legal_work_types`, `payment_tasks`, `approval_matrix`, `notifications`, `acceptance_tasks`, `avr_document_lines` и `avr_tasks`).

3. **Права доступа к расчетным представлениям (Views)**:
   Права на чтение (`SELECT`) представлений `contractor_project_balances_view` и `contractor_global_balances_view` были также предоставлены ролям `anon, authenticated`, при этом безопасность наследуется от базовых таблиц (так как представления созданы с опцией `security_invoker = true`).

## Альтернативы

### Альтернатива А: Полный запрет доступа для роли `anon`
- **Почему отклонена:** Клиентский SDK Supabase отправляет запросы к REST API с использованием роли `anon` на уровне соединения, прикрепляя JWT пользователя в заголовках. Полное отключение привилегий у роли `anon` ломает выборки для авторизованных пользователей.

## Последствия

### Положительные
- **Безопасность**: Доступ к бизнес-данным закрыт для неавторизованных пользователей на уровне RLS.
- **Работоспособность**: Исправлена проблема обнуления балансов и сумм AVR/выплат на страницах реестров.
- **Интеграция**: Корректно работают клиентские запросы через Supabase Client.

### Отрицательные
- При добавлении новых таблиц в миграциях Drizzle необходимо генерировать соответствующие RLS-политики и запускать скрипт `scripts/apply-rls.js` для актуализации прав.
