# ADR-16: Паттерн интеграции с 1С

**Статус:** Принято  
**Дата:** 2026-05-07  
**Связано с:** ADR-15 (AVR lifecycle), Мастер-роадмап

---

## Контекст

1С используется как система бухгалтерского и налогового учёта. MidesCloud — операционная система. Между ними существует несколько типов событий, которые должны быть синхронизированы:

**MidesCloud → 1С:**
- Подписанные АВР (для проводки расходов)
- Оплаченные счета поставщиков (для учёта кредиторской задолженности)
- Подтверждённые ЭСФ (для налогового вычета по НДС)

**1С → MidesCloud:**
- Статус проводки (расход посажен / отклонён)
- Факт получения ЭСФ (пришёл в ИС ЭСФ → 1С → MidesCloud)

---

## Принятое решение: Фазовая интеграция

Интеграция реализуется в три фазы по мере роста объёма операций.

---

### Фаза 1 (Сейчас): Ручной реестр

**Принцип:** MidesCloud — источник данных. Бухгалтер работает в обеих системах.

**MidesCloud → 1С:**
- Страница «Реестр для бухгалтерии»: список событий, требующих проводки
- Фильтр по периоду, типу события, статусу
- Экспорт: PDF (печатная форма) + CSV

```
Реестр содержит:
  - Подписанные АВР: контрагент, сумма, номер договора, период работ
  - Оплаченные счета: контрагент, сумма, дата, назначение
```

**1С → MidesCloud:**
- Бухгалтер вручную меняет статус в MidesCloud:
  - Нажимает «Посадить в учёт» → статус EXPENSE_POSTED
  - Вводит номер ЭСФ → статус ESF_RECEIVED

**Требования к реализации:**
- Роль `ACCOUNTANT` в системе прав
- Страница «Бухгалтерия» с реестром незакрытых событий
- Кнопки действий: «Принять АВР», «Отказать», «Зафиксировать ЭСФ», «Посадить»
- Лог действий бухгалтера (кто, когда, что сделал)

---

### Фаза 2 (6–12 месяцев): Файловый обмен

**Принцип:** Автоматизированный обмен файлами по расписанию.

**MidesCloud → 1С (автоэкспорт):**
```
Формат: CSV / Excel (совместим с импортом 1С)
Расписание: ежедневно в 08:00
Содержание: все события за предыдущий день
Доставка: сохранение в сетевую папку или email бухгалтеру
```

**1С → MidesCloud (автоимпорт):**
```
1С-разработчик настраивает выгрузку:
  - Список зарегистрированных ЭСФ (номер, контрагент, дата, сумма)
  - Список проведённых документов
Формат: CSV
MidesCloud читает файл по расписанию → автоматически обновляет статусы
```

**Требования к реализации:**
- Модуль импорта/экспорта с логом ошибок
- Маппинг контрагентов: ИИН/БИН для сопоставления между системами
- Уведомление бухгалтеру при ошибках импорта

---

### Фаза 3 (Будущее): REST API интеграция

**Принцип:** Real-time двусторонняя синхронизация через API.

**Технические детали:**
- 1С 8.3+ поддерживает HTTP-сервисы (REST/OData)
- MidesCloud вызывает 1С API при изменении статуса AVR
- 1С вызывает MidesCloud webhook при регистрации ЭСФ

**Когда переходить:**
- Объём операций > 100 AVR/месяц
- Задержка ручной синхронизации создаёт бизнес-проблемы
- Есть 1С-разработчик для настройки HTTP-сервисов

## Граница ответственности систем

```
MidesCloud (System of Engagement)        1С (System of Record)
─────────────────────────────────────    ──────────────────────────────
✅ Детальный учёт по проекту             ✅ Бухгалтерские проводки (ГК)
✅ Статусы АВР и ЭСФ                     ✅ Налоговый учёт (НДС, КПН)
✅ Субподрядный реестр                   ✅ Баланс и P&L по МСФО/НСФО
✅ Бюджет vs факт (operational)          ✅ Зарплата и табели
✅ Оперативный кеш-флоу                  ✅ Банковские выписки
❌ Юридические проводки                  ❌ Детальный учёт по проекту
❌ Налоговая отчётность                  ❌ Операционный workflow
```

**Событийная модель синхронизации:**

| Событие в MidesCloud | Действие в 1С |
|---|---|
| AVR_APPROVED | Бухгалтер создаёт акт в 1С |
| ESF_RECEIVED | Бухгалтер регистрирует счёт-фактуру в 1С |
| EXPENSE_POSTED | Подтверждение обратно в MidesCloud |
| payment.status = PAID | Бухгалтер проводит оплату в 1С |

---

## Идентификатор для маппинга

Для сопоставления контрагентов между системами используется **БИН/ИИН** (бизнес-идентификационный номер):

```sql
ALTER TABLE contractors ADD COLUMN bin TEXT UNIQUE; -- БИН для юрлиц
ALTER TABLE contractors ADD COLUMN iin TEXT UNIQUE; -- ИИН для физлиц
```

Это позволит автоматически сопоставлять записи при файловом обмене (Фаза 2).

## Последствия

**Положительные:**
- Фаза 1 реализуется в рамках P2P Phase 1 (минимальные усилия)
- Бухгалтер получает инструмент без необходимости изучать 1С детально
- Постепенная автоматизация без риска «всё или ничего»

**Отрицательные / риски:**
- Фаза 1: риск рассинхронизации при ошибках ручного ввода → решается логом и сверкой
- Фаза 2: зависимость от формата экспорта 1С — может потребовать 1С-специалиста
- БИН/ИИН нужно заполнить для всех существующих контрагентов (разовая работа)
