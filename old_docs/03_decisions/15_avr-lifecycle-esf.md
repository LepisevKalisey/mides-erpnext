# ADR-15: Жизненный цикл АВР с ролью бухгалтерии и ЭСФ

**Статус:** Принято  
**Дата:** 2026-05-07  
**Связано с:** ADR-11 (Dual-Path P2P), ADR-12 (Lifecycle)

---

## Контекст

В текущем ADR-12 статус `AVR_SIGNED` является конечной точкой для Payment Application пути. Это упрощение не отражает реальный процесс:

1. АВР создаётся в MidesCloud и передаётся в бухгалтерию
2. Бухгалтерия проверяет АВР: может принять или отказать
3. После принятия — ожидается ЭСФ (электронная счёт-фактура) от контрагента
4. ЭСФ регистрируется в 1С, бухгалтер делает проводку
5. Только после проводки расход считается «посаженным» — признанным для налогового учёта

Без этих состояний невозможно:
- Контролировать долги субподрядчиков корректно (ЭСФ не пришёл ≠ расход признан)
- Давать бухгалтерии инструмент работы внутри системы
- Интегрироваться с 1С в правильных точках

---

## Решение

### Расширенный lifecycle для Payment Application пути

```
SUBMITTED
  └─► WORK_CONFIRMED        — PM/прораб подтвердил объём работ
        └─► AVR_SUBMITTED   — АВР оформлен, передан в бухгалтерию
              ├─► AVR_REJECTED      — бухгалтерия отказала (с причиной)
              │     └─► SUBMITTED   — возврат на доработку
              └─► AVR_APPROVED      — бухгалтерия приняла, подписала
                    └─► ESF_PENDING — ожидаем ЭСФ от контрагента
                          └─► ESF_RECEIVED  — ЭСФ получен, зафиксирован
                                └─► EXPENSE_POSTED — проводка в 1С, расход посажен
```

### Таблица `avr` — дополнительные поля

```sql
ALTER TABLE avr ADD COLUMN accounting_reviewed_by UUID REFERENCES auth.users(id);
ALTER TABLE avr ADD COLUMN accounting_reviewed_at TIMESTAMPTZ;
ALTER TABLE avr ADD COLUMN rejection_reason TEXT;
ALTER TABLE avr ADD COLUMN esf_number TEXT;           -- номер ЭСФ из системы ИС ЭСФ
ALTER TABLE avr ADD COLUMN esf_received_at TIMESTAMPTZ;
ALTER TABLE avr ADD COLUMN expense_posted_at TIMESTAMPTZ;
ALTER TABLE avr ADD COLUMN expense_posted_by UUID REFERENCES auth.users(id);
```

### Роли в lifecycle

| Статус | Кто действует |
|---|---|
| SUBMITTED → WORK_CONFIRMED | PM / прораб |
| WORK_CONFIRMED → AVR_SUBMITTED | Снабженец / PM |
| AVR_SUBMITTED → AVR_APPROVED/REJECTED | **Бухгалтер** |
| AVR_APPROVED → ESF_PENDING | Автоматически |
| ESF_PENDING → ESF_RECEIVED | **Бухгалтер** (фиксирует номер ЭСФ) |
| ESF_RECEIVED → EXPENSE_POSTED | **Бухгалтер** (подтверждает проводку) |

### Карточка контрагента: АВР-реестр

Сводная таблица по всем АВР с данным контрагентом:

```
┌──────────────────────────────────────────────────────┐
│ ООО «СтройМонтаж» — АВР-реестр                       │
├─────────────────┬──────────┬────────────────────────┤
│ 🟢 Посаженные   │ 4 500 000₸ │ AVR + ЭСФ + проводка  │
│ 🟡 Ожидают ЭСФ  │   800 000₸ │ AVR одобрен           │
│ 🔵 На проверке  │ 1 200 000₸ │ У бухгалтерии         │
│ 🔴 Отказанные   │   150 000₸ │ Причина: нет акта     │
├─────────────────┴──────────┴────────────────────────┤
│ Итого к выплате (одобрено, без ЭСФ): 800 000₸       │
│ В процессе признания: 1 200 000₸                     │
└──────────────────────────────────────────────────────┘
```

### Важное правило

**Аванс субподрядчику блокируется, если сумма неоплаченных/непосаженных АВР превышает лимит.**

Логика: если у контрагента есть AVR_APPROVED (ожидает ЭСФ), это считается долгом системы перед ним. Новый аванс выдаётся только если предыдущие обязательства закрыты или в рамках утверждённого лимита.

---

## Альтернативы, которые не выбраны

- **Оставить AVR_SIGNED финальным:** теряем контроль над ЭСФ и налоговым признанием расходов
- **Вести АВР только в 1С:** теряем детальность по проекту и субподрядчику в операционной системе

---

## Последствия

**Положительные:**
- Бухгалтер получает рабочий инструмент в MidesCloud — не нужно дублировать в Excel
- Полная картина взаиморасчётов: что признано, что ожидает, что спорно
- Корректный контроль авансирования субподрядчиков
- Чёткая точка интеграции с 1С: только `EXPENSE_POSTED` события

**Отрицательные / риски:**
- Бухгалтер должен работать в MidesCloud — нужна простая и чёткая UX для этой роли
- Если ЭСФ не приходит долго → позиция «зависает» в ESF_PENDING → нужен SLA/напоминание
