# ADR-14: Overhead Project Type (Офис как проект)

**Статус:** Принято  
**Дата:** 2026-05-07  
**Связано с:** ADR-10 (Project Entity)

---

## Контекст

Офисные и административные расходы (аренда, коммунальные, оборудование) сейчас не привязаны ни к какому проекту. Это создаёт «серую зону»: расходы есть, но их нельзя отследить, сравнить с бюджетом или разделить по офисам при росте компании.

Альтернатива — создать отдельную логику для «неклиентских» расходов — приводит к двум разным путям создания заявок и усложняет систему.

---

## Решение

Ввести поле `type` в таблицу `projects` с тремя значениями:

```sql
ALTER TABLE projects ADD COLUMN type TEXT NOT NULL DEFAULT 'CONSTRUCTION';
-- Допустимые значения:
-- CONSTRUCTION  — строительный проект с клиентом и дедлайном
-- OVERHEAD      — административный/офисный (без дедлайна, годовой бюджет)
-- DEVELOPMENT   — внутренние инвестиции (будущее)
```

### Характеристики OVERHEAD-проекта

| Поле | CONSTRUCTION | OVERHEAD |
|---|---|---|
| `client_id` | Обязателен | NULL |
| `planned_end_date` | Обязателен | NULL (бессрочный) |
| `budget_period` | На проект | Годовой (Jan–Dec) |
| `phase` | BIDDING → ACTIVE → CLOSED | Всегда ACTIVE |
| P2P workflow | Полный | Полный (без изменений) |

### Стартовые OVERHEAD-проекты

При инициализации системы создаются автоматически:
- «Главный офис» (type=OVERHEAD) — дефолтный для офисных закупок
- При открытии нового офиса → создать новый OVERHEAD-проект

### Поведение в UI

- В селекторе проекта при создании заявки: OVERHEAD-проекты отображаются отдельной группой «Офисы / Административные»
- В аналитике: OVERHEAD и CONSTRUCTION разделены по умолчанию
- Сотрудник, привязанный к офису, видит соответствующий OVERHEAD-проект в списке доступных

---

## Альтернативы, которые не выбраны

- **Отдельная логика «административных закупок»:** двойной код, разные формы — отклонено
- **Тег/метка на позиции без проекта:** не даёт бюджетного контроля и аналитики по офису

---

## Последствия

**Положительные:**
- Единый P2P-воркфлоу для всех расходов — ноль специальных случаев
- Открытие нового офиса = создать один проект (2 минуты)
- Перевод сотрудника между офисами = изменить привязку к OVERHEAD-проекту
- Раздельный P&L по каждому офису из коробки

**Отрицательные / риски:**
- `planned_end_date` и `client_id` nullable — нужна валидация по типу проекта
- Бюджет нужно сбрасывать ежегодно для OVERHEAD — требует логики ежегодного обновления
