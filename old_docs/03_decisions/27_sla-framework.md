# ADR-027: SLA-фреймворк для процессов закупки и АВР

**Status:** Accepted  
**Date:** 2026-05-08  
**Релиз:** M1 P2P

---

## Context

Система должна измерять время выполнения каждого шага процесса и сигнализировать о нарушениях. Бесконтрольные очереди = задержки на объекте. Начальник снабжения должен видеть загрузку команды и перераспределять задачи. Директор видит сводную картину — не детали, а сигналы.

**Важно по срочным заявкам:** Срочная заявка — это следствие плохого планирования на участке. Снабжение информируется о срочности, но самостоятельно принимает решение о приоритетах. SLA для срочных заявок такой же — снабжение не наказывается за чужие просчёты.

## Нормативы SLA

### Закупочный процесс (Путь A — Procurement)

```
┌─────────────────────────────────────────────────────────────────┐
│  Шаг                   │  SLA      │  Владелец        │ Триггер │
├─────────────────────────────────────────────────────────────────┤
│  Утверждение заявки    │  3 раб.   │  DIRECTOR/DEPUTY │ PEND_   │
│                        │  дня      │                  │ APPROVAL│
├─────────────────────────────────────────────────────────────────┤
│  Назначение из пула    │  16 раб.  │  Пул снабжения   │ POOL_   │
│  (ничейная заявка)     │  часов*   │  (PROCUREMENT)   │ AVAIL.  │
├─────────────────────────────────────────────────────────────────┤
│  Выбор поставщика      │  3 раб.   │  PROCUREMENT     │ С момен-│
│  (после назначения)    │  дня      │  (назначенный)   │ та наз. │
├─────────────────────────────────────────────────────────────────┤
│  Утверждение PO        │  3 раб.   │  DIRECTOR/DEPUTY │ PO в    │
│                        │  дня      │                  │ PENDING │
└─────────────────────────────────────────────────────────────────┘

* 16 рабочих часов = если заявка поступила в 18:00, 
  к 9:00 следующего рабочего дня она должна быть назначена.
  Рабочее время: 09:00–18:00 пн–пт
```

**Максимальный цикл от заявки до PO:** 3+2+3+3 = ~11 рабочих дней

### АВР-процесс

```
┌─────────────────────────────────────────────────────────────────┐
│  Шаг                    │  SLA      │  Владелец                 │
├─────────────────────────────────────────────────────────────────┤
│  Назначение в АВР-пуле  │  16 раб. ч│  Рук. ПТО / ГИ           │
│  (ничейный АВР)         │           │                           │
├─────────────────────────────────────────────────────────────────┤
│  Проверка и «окунание»  │  2 раб.   │  PTO / GI / ACCOUNTING   │
│  каждым участником      │  дня      │                           │
├─────────────────────────────────────────────────────────────────┤
│  Финальное «окунание»   │  1 раб.   │  DIRECTOR                │
│  директором             │  день     │                           │
├─────────────────────────────────────────────────────────────────┤
│  ЭСФ от субподрядчика   │  30 кал.  │  ACCOUNTING              │
│  (мониторинг)           │  дней     │  (следит, не управляет)  │
└─────────────────────────────────────────────────────────────────┘
```

> **Терминология:** «Окунание» — термин, принятый внутри команды для обозначения подписания/согласования АВР участником пула. Используется в системе как UX-ярлык. В интерфейсе — кнопка «Согласовать».

## Модель отслеживания SLA

SLA вычисляется из уже существующей таблицы `purchase_events` (аудит-трейл), которая фиксирует каждый переход статуса.

### Таблица нормативов

```sql
CREATE TABLE midescloud.sla_config (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  step_name     TEXT NOT NULL UNIQUE,
  -- APPROVAL_WAIT     = ожидание утверждения
  -- POOL_ASSIGNMENT   = назначение из пула (16ч)
  -- SOURCING          = выбор поставщика
  -- PO_APPROVAL       = утверждение PO
  -- AVR_ASSIGNMENT    = назначение АВР в пуле
  -- AVR_REVIEW        = проверка участником пула
  -- AVR_DIRECTOR      = финальное согласование директором
  target_hours  NUMERIC NOT NULL,
  is_working_hours BOOLEAN NOT NULL DEFAULT true,
  -- true = только рабочее время 09:00-18:00 пн-пт
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Начальные значения SLA
INSERT INTO midescloud.sla_config (step_name, target_hours, is_working_hours) VALUES
  ('APPROVAL_WAIT',   72,  true),  -- 3 раб. дня × 9ч = 27ч, но ≈ 3 дня
  ('POOL_ASSIGNMENT', 16,  true),  -- 16 рабочих часов
  ('SOURCING',        72,  true),  -- 3 раб. дня
  ('PO_APPROVAL',     72,  true),  -- 3 раб. дня
  ('AVR_ASSIGNMENT',  16,  true),  -- 16 рабочих часов
  ('AVR_REVIEW',      36,  true),  -- 2 раб. дня × 9ч + буфер
  ('AVR_DIRECTOR',    9,   true);  -- 1 раб. день
```

### Расчёт нарушений (View)

```sql
-- Пример: заявки с просрочкой на шаге POOL_ASSIGNMENT
CREATE OR REPLACE VIEW midescloud.sla_breaches AS
SELECT
  pi.id                     AS item_id,
  pi.title,
  pi.status,
  pe.created_at             AS step_started_at,
  sc.target_hours,
  sc.step_name,
  -- упрощённый расчёт (без учёта рабочего времени для MVP)
  EXTRACT(EPOCH FROM (now() - pe.created_at)) / 3600 AS hours_elapsed,
  CASE
    WHEN EXTRACT(EPOCH FROM (now() - pe.created_at)) / 3600 > sc.target_hours
    THEN true ELSE false
  END                        AS is_breached,
  pi.assigned_to,
  pi.object_id
FROM midescloud.purchase_items pi
JOIN midescloud.purchase_events pe ON pe.purchase_item_id = pi.id
  AND pe.to_status = pi.status   -- последний переход = текущее состояние
JOIN midescloud.sla_config sc ON sc.step_name = CASE pi.status
  WHEN 'PENDING_APPROVAL' THEN 'APPROVAL_WAIT'
  WHEN 'POOL_AVAILABLE'   THEN 'POOL_ASSIGNMENT'
  WHEN 'IN_SOURCING'      THEN 'SOURCING'
  ELSE NULL END
WHERE pi.status IN ('PENDING_APPROVAL', 'POOL_AVAILABLE', 'IN_SOURCING');
```

## Управление загрузкой снабжения

### Роль: Руководитель отдела снабжения (PROCUREMENT_LEAD)

Это сотрудник с ролью `PROCUREMENT` и флагом `is_team_lead = true`.  
Дополнительные возможности сверх обычного снабженца:

| Действие | PROCUREMENT | PROCUREMENT_LEAD |
|---|:---:|:---:|
| Взять заявку из пула | ✅ | ✅ |
| Назначить заявку другому снабженцу | — | ✅ |
| Видеть все заявки команды (не только свои) | — | ✅ |
| Видеть загрузку каждого сотрудника | — | ✅ |
| Перераспределить задачи | — | ✅ |

## Уведомления при нарушении SLA

```
Нарушение шага POOL_ASSIGNMENT (16ч):
  → Уведомление PROCUREMENT_LEAD:
    «Заявка "[Арматура А-12]" в пуле уже 18 часов без назначения»

Нарушение шага APPROVAL_WAIT (3 дня):
  → Уведомление DIRECTOR/DEPUTY:
    «Заявка "[Кирпич M500]" ждёт утверждения 4 дня»
    (повторно каждые 24 часа)

Нарушение шага SOURCING (3 дня):
  → Уведомление PROCUREMENT_LEAD:
    «Позиция "[Цемент]" у Иванова А. в подборе 4 дня без результата»
```

Канал уведомлений: **Telegram** (Phase 2), **In-App** (Phase 1).

---

## Квартальный приоритет: ускорение АВР

В конце квартала (< 15 дней до окончания) SLA для АВР-процесса НЕ меняется,  
но дашборд директора выводит специальный виджет **«Квартальный сигнал»**  
с подсчётом АВР в процессе и потенциальной суммой к признанию расхода.

Снабженцы и ПТО могут видеть напоминание: «До конца квартала X дней».  
Административного давления нет — команда видит контекст и расставляет приоритеты самостоятельно.
