# ADR-13: Модель роутинга позиций закупки

**Статус:** Обновлено  
**Дата:** 2026-05-07 → 2026-05-13  
**Связано с:** ADR-11 (dual-path-p2p), ADR-12 (lifecycle), ADR-30 (object-as-child), ADR-33 (cascade-visibility), ADR-28 (material-intelligence)

---

## Контекст

После создания позиции закупки (`purchase_item`) она должна быть назначена ответственному за обработку. Нужен гибридный подход: автоматика для известных паттернов, пул для исключений.

---

## Решение (актуальное — май 2026)

**Источник правил маршрутизации: `project_assignments`**

Отдельная таблица `routing_rules` / `supply_auto_routing` **упразднена**.  
Правила маршрутизации берутся напрямую из `project_assignments`:

```
project_assignments WHERE project_role = 'BUYER'
```

Это исключает дублирование: команда проекта — это и есть правила роутинга.

---

## Алгоритм `routeItem()` при SUBMIT

```
function routeItem(item):

  1. IF item.flow_type == 'PAYMENT_APPLICATION':
       → Прямой путь. Назначить менеджеру договора. END.

  2. Поиск BUYER по объекту:
     candidates = project_assignments
       WHERE project_id = item.project_id
         AND project_role = 'BUYER'
         AND (object_id = item.object_id OR object_id IS NULL)
       ORDER BY object_id NULLS LAST  -- точное совпадение по объекту имеет приоритет

  3. Если candidates пусто → PENDING_SUPPLY (пул). END.

  4. Балансировка нагрузки среди кандидатов:
     a. ПРИОРИТЕТ: есть ли кандидат, у которого уже есть IN_PROGRESS позиция
        с тем же item.name (сравнение по названию)?
        → Назначить ему (он уже ищет аналог — экономия времени, контекст уже есть)
        Примечание: В Фазе 3 (ADR-28) name-matching заменится на сравнение по тегам
        материального каталога. До тех пор — точное совпадение по строке name.
     b. Нет совпадения → назначить кандидату с наименьшим числом позиций
        в статусе IN_PROGRESS (COUNT по purchase_items WHERE assigned_to_id = candidate)
     c. При равенстве нагрузки → isPrimary = true первым, затем по дате назначения

  5. Назначить: status = 'IN_PROGRESS', assigned_to_id = winner.user_id

  6. Все кандидаты-BUYER видят историю всех позиций проекта
     (через AllItemsTable с фильтром по проекту)
```

---

## Принцип каскадной видимости (см. ADR-33)

Если снабженец назначен на проект без привязки к конкретному объекту (`object_id IS NULL`) — он отвечает за **все объекты** этого проекта.  
Если назначен на конкретный объект — только за него.

```
Иерархия: Компания → Проект → Объект → Позиция
Правило: отсутствие ограничения на нижестоящий уровень = видимость всего нижестоящего
```

---

## Пул снабжения (fallback)

- Статус: `PENDING_SUPPLY`
- Видны всем снабженцам с ролью BUYER
- Любой может взять (`claimItem`) — переводит в `IN_PROGRESS`
- Все BUYER на проекте видят историю всех позиций через «Все позиции»

---

## Страница `/supply/routing`

**Упразднена.** Перенаправляет на `/projects` с пояснительным баннером.  
Управление распределением снабженцев — через вкладку «Команда» проекта.

---

## Последствия

**Положительные:**
- Единый источник истины: `project_assignments`
- Автобалансировка: тот, кто уже ищет аналог, получает похожую позицию
- Нет дублирующей таблицы routing_rules

**Отрицательные / риски:**
- Если снабженец не назначен на проект — позиция в пул (ожидаемое поведение)
- Балансировка COUNT(*) — лёгкий запрос, но при росте нагрузки можно кэшировать
