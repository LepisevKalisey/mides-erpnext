# ADR-11: Двухпутевой P2P-воркфлоу (Procurement vs Payment Application)

**Статус:** Принято  
**Дата:** 2026-05-07  
**Связано с:** ADR-10 (project-entity), ADR-12 (lifecycle), ADR-13 (routing)

---

## Контекст

Анализ выявил, что в системе смешаны два принципиально разных процесса:

| Процесс | Описание | Инициатор | Поставщик |
|---|---|---|---|
| **Закупка** | Нужда есть, поставщик неизвестен | Любой сотрудник | Ищет снабжение |
| **Заявка на оплату** | Договор есть, нужна оплата за выполненные работы | Менеджер по договору | Уже известен |

Смешивание этих процессов в одной форме приводит к:
- Неправильному роутингу (снабжение получает заявки, которые не нужно обрабатывать)
- Дублированию — один и тот же субподрядчик может быть «найден» снабжением повторно
- Нечёткому lifecycle (не ясно, что является следующим шагом)

---

## Решение

Вводится явное поле `flow_type` в `purchase_items`, определяющее путь обработки.

### Путь A: Procurement (Закупка)

**Используется когда:** нужда есть, поставщик неизвестен, нет активного договора.

```
Сотрудник создаёт позицию (PR)
  → flow_type = PROCUREMENT
  → Роутинг: автоматический (по правилам) или в пул
  ↓
Снабжение обрабатывает: запрашивает коммерческие предложения
  ↓
Создаётся Purchase Order (PO): поставщик + цена зафиксированы
  → Approval #2: руководитель/директор утверждает цену и PO
  ↓
Поставка: приёмщик подтверждает факт (GRN)
  → Three-Way Match: PR == PO == GRN
  ↓
Оплата: казначей выпускает платёж
```

**Применяется для:**
- Материалы (бетон, арматура, расходники)
- Услуги без договора (спецтехника по разовому запросу, новый субподрядчик)
- Офисные закупки

### Путь B: Payment Application (Заявка на оплату)

**Используется когда:** есть активный договор с субподрядчиком, работы выполнены, нужна оплата.

```
Менеджер создаёт позицию
  → flow_type = PAYMENT_APPLICATION
  → Привязка к contract_id (обязательна)
  → Роутинг: прямо на утверждение (минует пул снабжения)
  ↓
Approval: ответственный менеджер подтверждает объём выполнения
  ↓
Approval #2: директор утверждает сумму к оплате
  ↓
АВР: подписывается с субподрядчиком (= признание расхода)
  ↓
Оплата: казначей выпускает платёж
```

**Применяется для:**
- Авансы субподрядчику под договор
- Промежуточные оплаты по факту АВР
- Финальный расчёт по договору

### Логика выбора пути в форме

```
Тип = Материалы
  → flow_type = PROCUREMENT (всегда)

Тип = Услуги
  → Есть активный договор с этим субподрядчиком?
       ДА  → flow_type = PAYMENT_APPLICATION
       НЕТ → flow_type = PROCUREMENT (снабжение ищет исполнителя)
```

### Схема

```sql
ALTER TABLE purchase_items
  ADD COLUMN flow_type TEXT NOT NULL DEFAULT 'PROCUREMENT';
  -- PROCUREMENT | PAYMENT_APPLICATION

ALTER TABLE purchase_items
  ADD COLUMN contract_id UUID REFERENCES contracts(id);
  -- required when flow_type = PAYMENT_APPLICATION
```

---

## Альтернативы, которые не были выбраны

- **Одна форма для всего:** оставить текущий подход. Отклонено — смешивает роли снабжения и менеджеров контрактов, создаёт ложный роутинг.
- **Две полностью отдельные страницы:** возможно, но «Мои заявки» должны отображать оба типа в одном месте. Лучше один объект с разным поведением.

---

## Последствия

**Положительные:**
- Снабжение получает только те позиции, которые им реально нужно обрабатывать
- Менеджеры договоров не зависят от снабжения для текущих субподрядчиков
- «Мои заявки» = единое место для всех позиций любого типа
- Чёткий жизненный цикл для каждого пути

**Отрицательные / риски:**
- Форма должна адаптироваться под flow_type (разные поля, разные кнопки)
- При смене flow_type (пользователь передумал) — нужна логика сброса состояния

**Зависимые решения:**
- ADR-12: Lifecycle states (разные для каждого пути)
- ADR-13: Routing (правило 3 — Service + no contract → pool)
