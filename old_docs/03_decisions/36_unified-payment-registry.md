# ADR-36: Единый реестр казначея — объединение PROCUREMENT и PAYMENT_APPLICATION

**Статус:** Принято → Частично заменён конвеерной архитектурой (ADR Value Stream Map)
**Дата:** 2026-05-14
**Обновлено:** 2026-05-18
**Связано с:** ADR-11 (dual-path-p2p), ADR-12 (lifecycle), `docs/02_research/03_value-stream-map.md`

> **Примечание (2026-05-18):** Концепция «единого реестра казначея» как центральной
> сущности заменена конвеерной архитектурой. Казначей теперь видит оплаты как шаги
> в конвеерах L2-A (A1.6), L2-B (B6, B-ADV.3), L2-HR (HR.5). Группировка
> Контрагент → Проект сохраняется как UI-паттерн, но не как архитектурный якорь.
> Техническая реализация (WHERE status = 'APPROVED') остаётся рабочей.

---

## Контекст

ADR-11 определяет два пути P2P:
- **Путь A (PROCUREMENT):** закупка, поставщик неизвестен → снабжение → PO → утверждение директором
- **Путь B (PAYMENT_APPLICATION):** договор есть → оплата субподрядчику → утверждение директором

Ранее казначей видел **только PAYMENT_APPLICATION** позиции в статусе APPROVED.
PROCUREMENT позиции (утверждённая аренда техники, офисные закупки) были «невидимы» для оплаты.

## Решение

После утверждения директором **все** позиции статуса APPROVED поступают в единый реестр казначея
независимо от `flow_type`.

### Слияние потоков

```
Путь A: PROCUREMENT (PENDING_APPROVAL)   ──┐
                                            ├── Директор /approvals ──► status = APPROVED
Путь B: PAYMENT_APPLICATION (SUBMITTED)  ──┘
                                                          │
                                                          ▼
                                               Казначей /treasury
                                          Единый реестр: Контрагент → Проект → Позиции
                                                          │
                                                    status = PAID
```

### Группировка у казначея

```
Контрагент (уровень 1) — сколько всего надо заплатить
  └── Проект (уровень 2) — с какого счёта платить
        └── Позиции (листья) — что конкретно оплачивается
              [FlowBadge: АВР | Закупка]
```

Объект убран из иерархии: казначею важен контрагент и проект (банковский счёт), а не строительный объект.

### Технический фикс: contractor_id для PROCUREMENT

При утверждении PROCUREMENT позиции через `batchApproveRegistry`:
- `contractor_id` копируется из `sourcing_quotes.contractor_id` (выбранной котировки)
- `amount_gross` копируется из `sourcing_quotes.amount` если в заявке NULL

Это делает treasury query тривиальным — один фильтр `WHERE status = 'APPROVED'` без JOIN в sourcing_quotes.

## Последствия

**Положительные:**
- Казначей видит ВСЕ утверждённые к оплате позиции в одном месте
- Одна кнопка «Оплатить» для любого типа позиции
- Убрана вкладка «Закупки» (дублирование, пережиток)
- KPI strip считает реальный объём к оплате (оба пути)

**Отрицательные / риски:**
- PROCUREMENT позиции без выбранной котировки (`selected_quote_id = NULL`) будут без contractor_id после утверждения → не попадут в группировку по контрагенту → отобразятся в группе «Без контрагента»
