# ADR-10: Project как самостоятельная сущность (Вариант C)

**Статус:** Принято  
**Дата:** 2026-05-07  
**Связано с:** ADR-04 (contractor-hierarchy), ADR-11 (dual-path-p2p)

---

## Контекст

Текущая схема данных привязывает все финансовые операции к `objects` (физическим строительным объектам). Это создаёт следующие ограничения:

- Невозможно вести бюджет на уровне проекта (договор с заказчиком vs. физические объекты)
- Нет поддержки нескольких физических участков в рамках одного контракта
- Нет основы для сравнения факт/бюджет (Job Costing)
- Нет учёта жизненного цикла: тендер → строительство → закрытие

Рассматривались три варианта:

| Вариант | Описание | Проблема |
|---|---|---|
| A | Object = Project (текущее) | Один объект ≠ один контракт с заказчиком |
| B | Project является дочерним для Object | Инвертированная логика — нелогично |
| **C** | **Project — бизнес-сущность, Object — физическое место** | ✓ Принято |

---

## Решение

Вводится сущность `projects` как первоклассная бизнес-единица.

### Определения

**Project (Проект):**
- Коммерческая единица: договор с заказчиком, бюджет, сроки
- Может включать один или несколько физических объектов (участков)
- Примеры: «Жилой комплекс Мирас», «Бизнес-центр Алтын»

**Object (Объект):**
- Физическое место: адрес, площадка, участок
- Всегда принадлежит ровно одному проекту
- Остаётся для операционного учёта (за каким объектом закреплён сотрудник, какой роутинг применяется)
- Может принадлежать проекту или существовать без проекта (административный офис, хозяйственные закупки)

### Схема

```sql
CREATE TABLE projects (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name              TEXT NOT NULL,
  code              TEXT UNIQUE,              -- краткий код (МР-2024)
  client_id         UUID REFERENCES contractors(id),
  status            TEXT NOT NULL DEFAULT 'ACTIVE',
    -- BIDDING | ACTIVE | CLOSEOUT | CLOSED
  contract_value    DECIMAL(15, 2),           -- сумма договора с заказчиком
  budget_baseline   DECIMAL(15, 2),           -- утверждённый базовый бюджет
  start_date        DATE,
  planned_end_date  DATE,
  created_at        TIMESTAMPTZ DEFAULT now(),
  updated_at        TIMESTAMPTZ DEFAULT now()
);

-- Объект всегда принадлежит ровно одному проекту (один-ко-многим)
-- project_objects M2M не нужна — достаточно простого FK на объекте
ALTER TABLE objects
  ADD COLUMN project_id UUID REFERENCES projects(id);
  -- nullable: административные объекты (офис) существуют без проекта

-- Добавить FK в purchase_items (nullable для обратной совместимости)
ALTER TABLE purchase_items
  ADD COLUMN project_id UUID REFERENCES projects(id),
  ADD COLUMN cost_type  TEXT;
  -- MATERIAL | LABOR | EQUIPMENT | SUBCONTRACT | OVERHEAD
```

### Бюджетная модель с учётом «плохих проектов»

Проект имеет два значения бюджета:

```
budget_baseline   = исходный бюджет из сметы
budget_current    = baseline + Σ(утверждённые change_orders)
```

Позиция закупки создаётся даже без привязки к проекту (nullable). При наличии проекта система отображает предупреждение если нет бюджетной строки — но не блокирует операцию.

---

## Альтернативы, которые не были выбраны

- **Вариант B (Object → Project):** инвертирует логику. Клиент подписывает договор с компанией, а не с объектом.
- **Оставить objects как есть:** невозможно вести бюджет, нет job costing, проект не может охватывать несколько адресов.

---

## Последствия

**Положительные:**
- Фундамент для job costing (бюджет → факт по каждому проекту)
- Поддержка нескольких участков в одном проекте
- Корректный учёт субподрядных договоров на уровне проекта
- Возможность сравнивать плановые и фактические показатели
- Простая схема: FK вместо M2M-таблицы (нет лишней таблицы, нет дублирования)

**Отрицательные / риски:**
- Необходима миграция UI: селектор проекта в форме закупки (сначала проект, затем объект)
- Отчётность по объектам остаётся, но вторична по отношению к проекту
- Для офисных закупок (без проекта) — поле nullable, без ограничений

**Зависимые решения:**
- ADR-11: Dual-path P2P
- ADR-12: Lifecycle states (используют project_id)
- ADR-13: Routing model (routing_rules по project + work_type)
