# ADR-48: PM утверждает АВР после проверки ПТО

**Статус:** Accepted  
**Дата:** 2026-05-23  
**Контекст:** Gap Analysis выявил отсутствие PM-шага в workflow АВР. PM должен подтвердить коммерческую сторону АВР (в рамках договора и бюджета) после технической проверки ПТО.

---

## Контекст

Текущий AVR workflow:
```
Прораб (физ.приёмка) → ПТО (проверка объёмов) → ??? → Бухгалтерия (COMMITTED)
```

PM пропущен в цепочке. Но по бизнес-логике PM ДОЛЖЕН подтвердить:
1. АВР в рамках суммы договора (нет перерасхода бюджета)
2. Объёмы работ соответствуют ожиданиям по проекту
3. Нет дублирования с предыдущими АВР

## Решение

### Обновлённый маршрут АВР

```
[1] Прораб: физическая приёмка (acceptance_tasks)
[2] Загрузка документа АВР + OCR (avr_tasks: DRAFT → UPLOADED)
[3] ПТО: проверка объёмов и data_status (avr_tasks: UPLOADED → data_status=SUBMITTED)
[4] PM: утверждение коммерческой стороны (avr_tasks: → PM_APPROVED) ← НОВЫЙ ШАГ
[5] Бухгалтерия: проверка оформления (avr_tasks: → COMMITTED)
```

### Изменения в enum `avr_status`

```
Текущий: DRAFT | UPLOADED | APPROVED | REJECTED
Новый:   DRAFT | UPLOADED | PTO_VERIFIED | PM_APPROVED | ACCOUNTING_CHECKED | REJECTED
```

| Статус | Кто переводит | Что проверяет |
|--------|--------------|---------------|
| DRAFT | Система | АВР создан, документ не загружен |
| UPLOADED | Прораб/PM | Документ загружен, ждёт ПТО |
| PTO_VERIFIED | ПТО-инженер | Объёмы корректны, data_status=SUBMITTED |
| PM_APPROVED | PM проекта | В рамках договора и бюджета |
| ACCOUNTING_CHECKED | Бухгалтер | Оформление по закону |
| REJECTED | Любой из цепочки | Возврат на доработку |

### Новое поле в `avr_tasks`

```
pm_approved_by: text (nullable)    — user_id PM, который утвердил
pm_approved_at: timestamp (nullable) — когда утвердил
pm_comment: text (nullable)         — комментарий PM (обратная связь)
```

### Кто видит АВР на каком этапе

| Роль | Видит АВР со статусом | Действие |
|------|----------------------|----------|
| Прораб | DRAFT, UPLOADED (свои) | Загрузить документ |
| ПТО | UPLOADED | Проверить объёмы → PTO_VERIFIED / REJECTED |
| PM | PTO_VERIFIED | Утвердить → PM_APPROVED / REJECTED |
| Бухгалтер | PM_APPROVED | Проверить оформление → ACCOUNTING_CHECKED / REJECTED |

### Что PM проверяет (чеклист в UI)

1. ✅ Сумма АВР ≤ остаток по позициям договора
2. ✅ Объёмы не дублируют предыдущие АВР
3. ✅ Субподрядчик указан корректно
4. ✅ Нет расхождений с физ.приёмкой

Если PM отклоняет — обязательный комментарий (`pm_comment`), АВР возвращается прорабу.

## Альтернативы

### A. PM только наблюдает, не утверждает
**Отклонено.** PM несёт коммерческую ответственность за проект. Без его подтверждения невозможно контролировать бюджет.

### B. PM и ПТО проверяют параллельно
**Отклонено.** PM проверяет ПОСЛЕ ПТО, т.к. ему нужны подтверждённые объёмы для сравнения с договором.

### C. PM утверждает через commitment (как сейчас)
**Отклонено.** Commitment — это заявка на оплату (до работ). АВР — это подтверждение выполненных работ (после). Это разные этапы.

## Последствия

1. Enum `avr_status` расширяется: 4 статуса → 6 статусов
2. `avr_tasks` получает 3 новых поля (pm_approved_by/at/comment)
3. `/dashboard/requests` показывает PM-ам АВР в статусе PTO_VERIFIED
4. Нотификация PM при переходе в PTO_VERIFIED
5. Закрывает юзерстори 1.3 (PM утверждает АВР и даёт обратную связь)
6. Обновить инвариант #5 в Blueprint: маршрут = SITE_MANAGER → ПТО → PM → Бухгалтерия
