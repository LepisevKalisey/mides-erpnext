# ADR-019: Прямое указание директора (Executive Override)

**Status:** Accepted  
**Date:** 2026-05-07

## Context

Директор периодически даёт прямое устное указание на оплату: «Оплати 1М Ернару за школу».
Казначей должен исполнить платёж немедленно, без ожидания полного цикла согласования.
Документы (АВР, счёт) собирает параллельно ответственный за контрагента.

Это не обход системы — это легитимный экстренный режим с полным audit trail.

## Decision

### Поведение системы

1. Директор создаёт `purchase_request` (заявку на оплату) стандартным способом.
2. Система автоматически переводит её в статус `APPROVED` минуя согласование — именно потому что создатель имеет роль DIRECTOR.
3. Заявка **автоматически назначается** ответственному за контрагента (`contractor_managers.user_id`).
4. Казначей видит её в своём списке «К оплате».
5. Ответственный видит её в своём списке с задачей «Собрать документы».

### Ключевое решение

**Не добавляется** отдельный флаг `is_direct_order` или колонка `direct_order_by`.

Поведение реализуется через:
- Роль создателя: `session.user.role === 'DIRECTOR'` → автоматический переход в `APPROVED`
- Audit log: `purchase_events` фиксирует `performed_by_id = director_id`, `event_type = 'status_change'`, `new_status = 'APPROVED'`
- По событию лога система знает, что это директорское решение

### Жизненный цикл

```
[Директор создаёт заявку]
        │
        ▼
   APPROVED          ← автоматически, т.к. создатель = DIRECTOR
        │
        ├── Казначей → PENDING_PAYMENT → PAID
        │
        └── Ответственный → собирает документы (PENDING_DOCS → DOCS_UPLOADED)
```

Документальное закрытие обязательно — без документов позиция остаётся с флагом незакрытой задачи.

### Назначение ответственного

```typescript
// При создании директором → assigned_to = contractor_managers.user_id
const manager = await db.query.contractor_managers.findFirst({
  where: eq(contractor_managers.contractor_id, request.contractor_id)
})
// Если менеджер не назначен → уведомление директору
```

## Alternatives Considered

- Отдельный флаг `is_direct_order` → избыточность, логика заложена в роли и audit log
- Отдельная таблица `direct_payment_orders` → усложнение схемы без выигрыша
- Принудительный полный цикл для директора → противоречит бизнес-требованию

## Consequences

- **+** Нет изменений в схеме данных
- **+** Директор не тормозится бюрократией в экстренных ситуациях
- **+** Полный audit trail через `purchase_events`
- **+** Ответственный автоматически получает задачу по документам
- **−** Ответственный должен быть назначен в `contractor_managers` — иначе некому назначить задачу
