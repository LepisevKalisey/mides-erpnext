# ADR-30: Объект как дочерняя сущность проекта

> [!NOTE]
> **УТОЧНЕНИЕ ПО БД ДЛЯ ВЕРСИИ V3**
> В соответствии с физической схемой базы данных версии v3 (`web/src/db/schema/projects.ts`), таблица объектов строительства называется **`project_objects`** (вместо `objects` во избежание конфликтов ключевых слов SQL и повышения читаемости). Все внешние связи ссылаются на `project_objects.id`. Концептуальное отношение (один проект ко многим объектам с непосредственной ссылкой `project_id` в таблице объекта) осталось без изменений.

**Статус:** Принято  
**Дата:** 2026-05-11  
**Заменяет:** ADR-10 (частично) — уточняет модель данных для объектов

---

## Контекст

В предыдущей реализации объекты (`objects`) существовали как самостоятельные сущности
в справочнике. Связь с проектом реализовывалась через junction-таблицу `project_objects`.

Это создавало архитектурное противоречие: объект не имеет смысла без проекта,
но модель данных допускала его существование вне проекта.

---

## Решения

### R-01: Объект — дочерняя сущность проекта

```
Старая модель:
objects (standalone) ←──── project_objects (junction) ────→ projects

Новая модель:
projects ──────────────────→ objects (project_id NOT NULL)
```

**Объект** = физическая единица строительства в рамках одного конкретного проекта.

Примеры:
```
Проект: «ЖК Мирас»
├── Объект: Корпус А  [CIVIL]
├── Объект: Корпус Б  [CIVIL]
├── Объект: Паркинг   [CIVIL]
└── Объект: Наружные сети [MEP]
```

**Правило**: один объект принадлежит ровно одному проекту.
Если та же физическая площадка фигурирует в другом договоре — это **новый объект** нового проекта.

### R-02: Ликвидация junction-таблицы project_objects

Таблица `project_objects` упраздняется.
Вместо неё: поле `project_id NOT NULL` непосредственно в таблице `objects`.

**Направление работ** (`work_stream`) — атрибут объекта (не связи):
- `CIVIL` — Общестрой
- `MEP` — Инженерные сети

### R-03: Ликвидация автономной страницы /objects

Страница `/objects` как самостоятельный раздел навигации упраздняется.

Управление объектами происходит **только внутри страницы проекта**:
- Просмотр: `/projects/[id]` → таб «Объекты»
- Создание: sidebar-шит «Добавить объект» на странице проекта
- Редактирование: inline на странице проекта

**Исключение**: admin-only агрегированный список объектов может существовать
для целей снабжения/отчётности, но не как основной UI.

### R-04: project_assignments — несколько пользователей на одну роль

Таблица `project_assignments`:
- **Старый уникальный ключ**: `(project_id, project_role)` — один человек на роль
- **Новый уникальный ключ**: `(project_id, user_id, project_role)` — несколько людей на роль

**Обоснование**: если проект разрастается, второй PM (или второй PTO_ENGINEER)
получает полный доступ наравне с первым. Логика OR: достаточно одного подтверждения
от любого носителя роли для прохождения workflow-gate.

## Последствия

### Изменения схемы БД

| Таблица | Действие |
|---------|----------|
| `project_objects` | DROP TABLE |
| `objects` | ADD COLUMN `project_id uuid NOT NULL REFERENCES projects(id)`, ADD COLUMN `work_stream work_stream` |
| `objects` | REMOVE COLUMN `status` (заменён статусом проекта) |
| `objects` | REMOVE UNIQUE на `name` (одно имя объекта в разных проектах допустимо) |
| `contracts` | `object_id` → ссылается на новую `objects.id` (FK остаётся) |

### Изменения в UI/маршрутизации

| Удалить | Заменить на |
|---------|-------------|
| `/objects` (страница) | `/projects/[id]` → таб «Объекты» |
| «Объекты» в сайдбаре | «Проекты» → `/projects` (уже есть) |

### Изменения в бизнес-логике

- Создание объекта: всегда требует `project_id`
- Запросы на материалы/АВР: объект выбирается из объектов конкретного проекта (не глобального справочника)
- Назначение начальника участка: через `project_assignments` с `object_id` (конкретный объект)
