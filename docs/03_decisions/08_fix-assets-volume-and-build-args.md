# Decision Record: Решение проблемы с раздачей ассетов (CSS/JS) и долгой сборкой при смене переменных в Coolify

- **Статус:** PROPOSED
- **Дата:** 2026-05-28
- **Автор:** World's Best DevOps Engineer

## Контекст

При деплое ERPNext v16 на Coolify возникли две критические проблемы:
1. **Отсутствие стилей (CSS/JS 404):** Сайт успешно запускается, но при обращении к CSS/JS ассетам Nginx возвращает ошибку 404. Это происходит потому, что каталог `/home/frappe/frappe-bench/sites` монтируется как Docker-volume для персистентности данных. Монтирование тома скрывает директорию `sites/assets`, скомпилированную в образе на этапе сборки. Скрипт `entrypoint.sh` пытается создать симлинк `sites/assets` -> `/home/frappe/frappe-bench/assets`, но в образе по этому пути ничего нет, так как `bench build` компилирует файлы напрямую в `sites/assets`.
2. **Медленный редеплой (30 минут):** Изменение любого параметра (например, URL-адреса хоста `SERVICE_URL_ERPNEXT`) в панели Coolify приводит к тому, что Coolify передает новые параметры как build arguments (`--build-arg`) Docker-демону. Это полностью инвалидирует Docker-кэш на самом первом шаге, заставляя заново скачивать все приложения и компилировать ассеты.

## Решение

Для устранения этих проблем предлагается внести следующие изменения (без выполнения коммитов и пушей до команды пользователя):

### 1. Исправление раздачи статики (CSS/JS)
Внести изменения в процесс сборки и запуска контейнера, чтобы отделить скомпилированные ассеты от монтируемого тома:
- **В [Dockerfile](file:///c:/Projects/Mides/ERPNext/Dockerfile):** После успешного завершения `bench build` копировать скомпилированные ассеты (каталог `sites/assets`) во внутренний каталог образа `/home/frappe/frappe-bench/assets`, который находится вне зоны действия тома. Также копировать файл `sites/apps.txt` в `/home/frappe/frappe-bench/apps.txt`.
- **В [entrypoint.sh](file:///c:/Projects/Mides/ERPNext/entrypoint.sh):** При старте контейнера восстанавливать актуальный список приложений (`cp -f /home/frappe/frappe-bench/apps.txt sites/apps.txt`) и создавать симлинк с примонтированного тома `sites/assets` на `/home/frappe/frappe-bench/assets` внутри образа. Это позволит Nginx отдавать актуальные JS/CSS файлы напрямую из образа, не затирая их томом.

### 2. Ускорение пересборки в Coolify (до 1-2 минут)
- **В панели Coolify:** Настроить переменные окружения (`SERVICE_URL_ERPNEXT`, `SERVICE_FQDN_ERPNEXT`, `DB_ROOT_PASSWORD` и др.) как **Runtime only** (убрать галочку "Build variable"). Это исключит их передачу в качестве build-arg и предотвратит инвалидацию кэша Docker при их изменении.
- **В [Dockerfile](file:///c:/Projects/Mides/ERPNext/Dockerfile):** Добавить использование BuildKit кэш-моунтов (`--mount=type=cache`) для pip-зависимостей (`/home/frappe/.cache/pip`) и yarn-зависимостей (`/home/frappe/.cache/yarn`). Это сохранит кэш пакетов на диске VPS даже в случае изменения кода или структуры Dockerfile, сокращая чистую установку до пары минут.

## Альтернативы Considered

- **Отдельный контейнер для раздачи статики (Nginx):** Отклонено. В текущих условиях лимитов ресурсов (8GB RAM на VPS) монолитный контейнер (Nginx + Supervisord + Gunicorn) является наиболее экономичным и простым решением, так как не требует запуска дополнительного контейнера и настройки совместных томов.
- **Монтирование каталога `assets` напрямую:** Невозможно из-за архитектурных ограничений Frappe, где Nginx ищет ассеты внутри пути `sites/assets`, который перекрывается общим томом `sites`.

## Последствия

- **CSS/JS ассеты заработают:** Симлинк будет указывать на реально существующую в образе директорию с ассетами, и Nginx сможет корректно отдавать файлы.
- **Моментальный редеплой при смене env:** Смена настроек/URL в Coolify больше не будет инвалидировать кэш Docker. Повторный деплой будет занимать около 10-15 секунд.
- **Быстрая сборка при изменении кода:** Использование кэш-моунтов BuildKit сократит время полной пересборки (например, при добавлении новых приложений) с 30 минут до 2-3 минут.
