# Decision Record: Перенос сборки в Coolify и оптимизация кэширования слоев Docker

- **Статус:** APPROVED
- **Дата:** 2026-05-28
- **Автор:** World's Best DevOps Engineer

## Контекст

После перехода на монолитную архитектуру время сборки в GitHub Actions составило **32 минуты**. Любое изменение конфигурационных файлов (`nginx.conf`, `supervisord.conf`, `entrypoint.sh`) полностью инвалидировало кэш и приводило к повторному скачиванию всех 11 приложений (`bench get-app`) и сборке ассетов (`bench build`).

Для решения этой проблемы было принято решение перенести сборку непосредственно в Coolify и отказаться от GitHub Actions, следуя только официальным и стандартным подходам Docker/ERPNext без усложнения архитектуры.

## Решение

1. **Отказ от GitHub Actions:**
   - Полностью удалены файлы рабочих процессов CI/CD (`.github/workflows/deploy.yml` и `.github/workflows/deploy-base.yml`).
   - Сборка образа приложения перенесена напрямую на сам сервер Coolify (Riga VPS) из Git-репозитория.

2. **Оптимизация Docker-кэширования (нативный подход):**
   - Файл `Dockerfile` реструктурирован согласно правилу "frequently changing instructions at the bottom" (часто меняющиеся инструкции внизу).
   - Команды `COPY nginx.conf`, `COPY supervisord.conf` и `COPY entrypoint.sh` перенесены в самый конец файла — **после** тяжелых шагов `bench get-app` и `bench build`.
   - При сборке на сервере Coolify локальный Docker-демон сохраняет кэш предыдущих слоев на диске VPS. Изменение конфигураций (Nginx, Supervisor, entrypoint) теперь собирается за **1–2 минуты**, так как Docker просто повторно копирует конфиги и пропускает скачивание приложений.

3. **Решение проблемы с сетью в Coolify:**
   - В `docker-compose.coolify.yml` не прописываются жесткие внешние сети.
   - Coolify автоматически создает внутреннюю сеть для стека и подключает контейнеры к своему прокси-серверу (Traefik) на порту 8080.
   - Если служба зависает в статусе `exited`, рекомендуется запустить **Redeploy** (или "Force Redeploy") в интерфейсе Coolify для автоматической пересборки контейнеров и обновления сетевых привязок.

## Альтернативы

- **Создание кастомного базового образа (Base Image):** Отклонено как избыточно сложное в обслуживании решение, требующее управления дополнительными репозиториями, сборщиками и реестрами образов.
- **Использование Docker Buildx Cache Mounts в GHA:** Отклонено из-за нестабильности кэширования GitHub Actions и необходимости его принудительного прогрева.

## Последствия

- **Простота:** В репозитории остался один стандартный `Dockerfile` и никаких CI/CD workflows.
- **Скорость:** Скорость отладки конфигураций на сервере сократилась до 1–2 минут за счет нативного кэширования локального Docker-демона на Riga VPS.
- **Соответствие стандартам:** Использование только официальных рекомендаций Docker и Frappe по оптимизации сборки.
