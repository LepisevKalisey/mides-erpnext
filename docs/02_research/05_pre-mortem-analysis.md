# Pre-mortem анализ сборки и деплоя ERPNext (MidesCloud)

**Статус:** ACTIVE  
**Дата:** 2026-05-27  
**Автор:** World's Best DevOps Engineer  

Pre-mortem — это метод, при котором мы мысленно переносимся в будущее, где наша система полностью отказала, и пытаемся понять, почему это произошло. Ниже представлены ключевые зоны риска для текущей инфраструктуры ERPNext на сервере с Coolify.

---

## 🚨 Топ-4 критических точек отказа (Где упадет в первую очередь?)

### 1. Нестабильные внешние ветки приложений (Билд Docker-образа)
**Как это сломается:**  
В `Dockerfile` установка кастомных приложений привязана к веткам `develop` или `version-16` сторонних репозиториев:
*   `clefincode_chat` (ветка `develop`)
*   `mint` (ветка `develop`)
*   `saas_theme` (ветка `version-16`)
*   `insights`, `pwa_frappe`, `eps`, `wiki` (без жестких версий)

Ветки `develop` постоянно обновляются разработчиками приложений. В любой момент коммит в репозиторий одного из этих приложений может:
1.  Сломать совместимость с Frappe v16.
2.  Привести к падению `pip install` или `yarn build` из-за несовместимых зависимостей.
3.  Репозиторий может быть удален или переименован.

**Последствие:** GitHub Actions перестанет собирать образ, деплой новых версий заблокируется.

---

### 2. Исчерпание оперативной памяти (OOM Killer) на сервере 8GB RAM
**Как это сломается:**  
Frappe/ERPNext — крайне ресурсоемкий стек. Текущая конфигурация запускает **11 контейнеров**:
*   `backend` (Gunicorn web-server)
*   `frontend` (Nginx)
*   `websocket` (NodeJS)
*   `scheduler` (Frappe schedule runner)
*   `queue-short` и `queue-long` (Celery-подобные воркеры Frappe)
*   `db` (MariaDB 11.8)
*   `redis-queue` и `redis-cache` (Redis)
*   Временные `configurator` и `create-site`

При росте базы данных или активном использовании (особенно при фоновой генерации PDF через Print Designer, синхронизации с банком в Mint или обработке аналитики в Insights) потребление RAM выйдет за пределы 8 ГБ. Линуксовый `OOM Killer` начнет принудительно убивать процессы. Первым кандидатом на убийство обычно становится `mariadb` (как самый прожорливый процесс).

**Последствие:** База данных падает со статусом `exited` (код 137), сайт выдает 500/503 ошибку, данные могут быть повреждены при аварийном завершении транзакций.

---

### 3. Ошибки миграции БД (`bench migrate` при деплое)
**Как это сломается:**  
При каждом обновлении контейнер `create-site` выполняет `bench --site frontend migrate`. 
Если обновление одного из приложений (например, HRMS или Insights) меняет схему базы данных (DocTypes) так, что это приводит к конфликту с уже существующими данными строительной компании, миграция завершится ошибкой.

**Последствие:** 
Поскольку у контейнера `create-site` стоит политика `restart_policy: none`, падение миграции приведет к завершению контейнера со статусом `exited (1)`. При этом миграция не применится, база данных останется в промежуточном состоянии, а бэкенд продолжит работать на старой схеме данных, что приведет к массовым ошибкам `OperationalError` (missing column, table doesn't exist) при работе пользователей.

---

### 4. Блокировка лимита подключений MariaDB (`max_connections`)
**Как это сломается:**  
Каждый воркер очереди (`queue-short`, `queue-long`), шедулер, бэкенд-воркеры Gunicorn и внешние API-интеграции держат постоянные подключения к MariaDB. По умолчанию в MariaDB лимит подключений равен 151. Если нагрузка возрастет или фоновые задачи начнут плодить процессы, лимит будет исчерпан.

**Последствие:** В логах бэкенда появится ошибка `OperationalError: (1040, "Too many connections")`, сайт перестанет отвечать на любые запросы.

## 🛠️ План превентивных мер (Как это предотвратить?)

| Зона риска | Решение | Статус |
|:---|:---|:---|
| **Билд образов** | Зафиксировать коммиты (SHA) или стабильные теги вместо веток `develop` в Dockerfile для сторонних приложений. | ⏳ Рекомендуется |
| **Ресурсы (RAM)** | Настроить swap-файл на сервере (минимум 4 ГБ) для компенсации пиков нагрузки, ограничить RAM для контейнеров Redis и MariaDB в compose-файле. | ⏳ Рекомендуется |
| **Мониторинг** | Настроить Healthcheck в Coolify для всех контейнеров, чтобы сломанные воркеры перезапускались автоматически. | ⏳ Рекомендуется |
| **Пароли** | Избегать использования символов `$`, `\`, `"`, `'` в `DB_ROOT_PASSWORD`, чтобы не ломать экранирование в скрипте GRANT. | ⏳ Рекомендуется |

## 🛠️ Дополнение: Pre-mortem интеграции Nginx в монолит (Решение проблемы CSS)

Мы возвращаем Nginx внутрь монолитного контейнера для раздачи статических ассетов (CSS, JS) и проксирования запросов на Gunicorn (порт 8000).

### 🚨 Возможные точки отказа плана и меры их предотвращения:

1. **Падение Nginx из-за прав доступа (Permission Denied):**
   * *Угроза:* По умолчанию Nginx работает от `root` и пытается писать логи в `/var/log/nginx/` и PID в `/run/nginx.pid`. При запуске от пользователя `frappe` в Supervisord он упадет.
   * *Решение:* В `Dockerfile` мы выполняем `chown -R frappe:frappe /var/log/nginx /var/lib/nginx /run /etc/nginx`. В кастомном `nginx.conf` PID-файл вынесен в `/home/frappe/nginx.pid`, а логи перенаправлены в `/dev/stdout` и `/dev/stderr`.

2. **Конфликт портов 8080 (Address already in use):**
   * *Угроза:* Если Gunicorn останется на порту `8080`, а Nginx попытается занять тот же порт `8080` для внешнего трафика, один из процессов упадет.
   * *Решение:* Мы строго разводим порты: Gunicorn настраивается на локальный порт `127.0.0.1:8000` (через `supervisord.conf` и `entrypoint.sh`), а Nginx занимает публичный порт `0.0.0.0:8080`.

3. **Ошибки раздачи статики через символические ссылки:**
   * *Угроза:* В Frappe папка `sites/assets` является симлинком на `assets`. Если Nginx не сможет перейти по ссылке из-за ограничений прав, CSS выдаст `403 Forbidden`.
   * *Решение:* Права на всю директорию `/home/frappe` рекурсивно принадлежат пользователю `frappe`. В `nginx.conf` путь к корню сайтов настроен стандартно, без использования `alias`, что гарантирует беспрепятственное разрешение симлинков.

4. **Проблема CRLF (Windows-переводы строк) в `nginx.conf`:**
   * *Угроза:* При редактировании файлов на Windows-машине переводы строк могут преобразоваться в CRLF, что вызовет синтаксическую ошибку Nginx при парсинге конфига в Linux.
   * *Решение:* В `Dockerfile` добавлен шаг очистки CRLF для конфига: `RUN sed -i 's/\r$//' /etc/nginx/nginx.conf`.
