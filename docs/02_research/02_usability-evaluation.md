# Анализ юзабилити и соответствия ERPNext бизнес-требованиям МиДЭС

**Статус:** ACTIVE  
**Дата:** 2026-05-27  
**Тип:** Explanation / Research  

---

## Введение

Цель данного исследования — оценить юзабилити и функциональную применимость ERPNext v16 для операционных и финансовых процессов строительной компании МиДЭС (Казахстан). 

Оценка проводилась по результатам сквозного тестирования 4 ключевых сценариев:
1. **Закупки материалов (P2P):** Заявка прораба → Утверждение директором → Заказ поставщику → Поступление на склад → Счёт от поставщика → Оплата.
2. **Субподряд:** Заказ на строительно-монтажные работы (СМР) → Утверждение → Авансовый платёж (30%) → Закрытие актом.
3. **Бюджетный контроль:** Блокировка заказов при превышении бюджета проекта.
4. **Управленческая отчётность:** Accounts Payable, Stock Balance, Purchase Analytics, Budget Variance.

---

## 1. Функциональное соответствие строительным задачам

По результатам автоматического и ручного прогона сценариев, базовая функциональность ERPNext закрывает потребности компании на **85-90%**:

| Процесс | Реализация в ERPNext | Оценка | Комментарий |
| :--- | :--- | :--- | :--- |
| **Сквозной P2P** | Полная цепочка от `Material Request` до `Payment Entry` | **10 / 10** | Отличная прослеживаемость связей между документами. Автоматическое обновление статусов. |
| **Субподряд** | Учёт услуг без списания материалов на склад | **8 / 10** | Удобно оформляется через `Purchase Order` без stock-флага. Не хватает нативного удержания гарантийных сумм (требуется кастомизация счетов). |
| **Бюджетирование** | Контроль в разрезе Проектов и Статей Затрат | **9 / 10** | Блокировка работает жестко. Важно: для материальных позиций бюджет проверяется в момент списания (или привязывается к счетам затрат в строке заказа). |
| **Multi-Company** | Работа ТОО + 3-5 ИП в единой базе | **10 / 10** | Идеальная изоляция данных при общей сквозной аналитике и возможности консолидированной отчетности. |

## 2. Анализ юзабилити (Удобство использования по ролям)

Главное противоречие ERPNext — баланс между **мощным бэкендом** и **сложным фронтендом**. Ниже приведена оценка удобства интерфейса для различных ролей:

### 👷 Линейный персонал (Прораб, Складовщик, Инженер ПТО)
* **Юзабилити:** **2 / 10** (Критически неудобно)
* **Проблемы:**
  - Огромное количество полей ввода (налоги, склады, валюты, счета учета, скидки), которые прорабу не нужны и непонятны.
  - Высокий риск опечаток: выбор не того склада или контрагента ломает управленческий учет.
  - Интерфейс не оптимизирован под мобильные телефоны на стройплощадке.
  - Поиск материалов в справочнике (`Item`) перегружен деталями.

### 💼 Офисный персонал (Снабженец, Экономист, Бухгалтер, Казначей)
* **Юзабилити:** **7 / 10** (Приемлемо)
* **Плюсы:**
  - Удобные реестры документов с фильтрацией и цветовым кодированием статусов.
  - Легкий доступ к истории изменений каждого документа.
  - Быстрый импорт данных из Excel.
* **Минусы:**
  - Терминология ERPNext (даже при русской локализации) требует привыкания (например, `Material Request` = Заявка на материал, `Purchase Receipt` = Приёмная накладная).

### 👑 Руководство (Директор, Руководитель проекта)
* **Юзабилити:** **8 / 10** (Хорошо)
* **Плюсы:**
  - Удобный механизм согласования (Workflows) прямо из веб-версии или почтовых уведомлений.
  - Наличие готовых дашбордов и отчетов в реальном времени.

## 3. Специфика Казахстана и локализации

1. **Валюта и БИН:** KZT поддерживается нативно. БИН/ИИН юрлиц легко добавляются через `Custom Fields` (Кастомные поля) без изменения ядра системы.
2. **Формы КС-2, КС-3, Р-1:** Стандартная ERPNext не знает этих форм. Решается созданием печатных форм (`Print Formats`) на HTML/Jinja в рамках кастомного приложения `mides_construction`.
3. **Русский язык:** Локализация покрывает 95% интерфейса Desk. Оставшиеся 5% (специфичные термины модулей) переводятся через инструмент перевода (`Translation`) прямо в базе данных.

## 4. Итоговые выводы и рекомендации

### Плюсы внедрения ERPNext:
1. **Быстрый старт (Time-to-Market):** Мы развернули полноценную систему с настроенной базой данных, пользователями, ролями, бюджетами и отчетами менее чем за неделю. Кастомная разработка (MidesCloud v3) потребовала бы месяцев работы над аналогичной функциональностью бэкенда.
2. **Надежность данных:** Frappe Framework жестко контролирует ссылочную целостность данных (нельзя удалить проект, если по нему есть закупки) и логику двойной записи в бухгалтерии.

### Главный риск:
* **Непринятие системы прорабами.** Если заставить линейных сотрудников заполнять стандартные формы ERPNext, это приведет к саботажу и недостоверности данных.

### Рекомендация (Стратегия "Двухскоростной ERP"):
Для обеспечения максимального юзабилити и сохранения преимуществ ERPNext рекомендуется разделить интерфейсы:

```mermaid
graph TD
    subgraph Поле (Стройплощадка)
      A[Прораб] -->|Микро-UI: 3 поля / Telegram| B(API Gateway)
      C[Инженер ПТО] -->|Микро-UI: Ввод объемов| B
    end
    
    subgraph Бэк-Офис (Управление)
      B -->|REST API| D[(ERPNext Core)]
      E[Снабженец] -->|ERPNext Desk| D
      F[Бухгалтер] -->|ERPNext Desk| D
      G[Директор] -->|Approve / Dashboards| D
    end
```

1. **Бэк-энд и Бэк-офис:** ERPNext выступает в качестве единого ядра данных (Single Source of Truth), где работают снабженцы, бухгалтеры и директор.
2. **Линейный интерфейс:** Вместо полной разработки ERPNext с нуля, ресурсы разработки (Next.js 14) направляются на создание **ультра-простого мобильного личного кабинета** для прорабов, который через REST API создает документы `Material Request` и `Purchase Receipt` в ERPNext.
